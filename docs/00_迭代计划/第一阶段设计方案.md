# 第一阶段设计方案

本文档根据 [迭代计划与排期说明](迭代计划与排期说明.md) 与 [第一阶段设计符合性检查](第一阶段设计符合性检查.md) 中的**第一阶段（鑫智链自营业务 MVP）**要求，整理为可实施的具体设计方案，供开发与验收对齐。执行动作与验收清单见 [设计落地执行方案](设计落地执行方案.md)。

---

## 一、目标与范围

### 1.1 阶段目标

- **交付节点**：3 月 27 日自营业务 MVP 上线，可运营、可验证。
- **核心价值**：为鑫智链搭建可运营的、核心功能闭环的产品碳足迹数据服务平台，完成订单 → 任务 → 报告 全链路及三端（中心端、供应商端、核查机构端）协同。

### 1.2 范围边界

| 类别 | 第一阶段范围 | 排除（后续迭代） |
|------|--------------|------------------|
| 平台基础 | 用户、权限、订单、任务管理；RBAC 与操作日志 | 结算中心（预留二迭代）；OA 电子签章（二迭代） |
| 中心端 | 驾驶舱（简版）、订单管理、任务管理、报告管理、供应商管理、核查机构管理、模板中心、系统管理 | 驾驶舱「二迭代做整体」；与鑫采商城/LCA 为对接预留 |
| 供应商端 | 工作台、待办任务、任务填报、待澄清处理、我的报告、企业信息与证照、知识库与帮助 | 绿色生态市场（P2 占位） |
| 核查机构端 | 核查仪表盘、核查任务管理、报告/证书颁发管理 | 穿透查看可为简化版；结算/机构管理为 P1 |
| 连接器 | 中心连接器**基础版**（轻量框架、服务注册与代理雏形） | **本阶段不用于生产**，仅技术储备 |

与 TDS 的身份、应用、接入连接器互联互通在设计中覆盖，实现为预留/对接，实施期以 [11_接口与互操作设计（实施期）](../02_功能与对接/11_接口与互操作设计（实施期）.md) 与 [22_TDS 架构与合规设计说明](../05_标准与合规/22_TDS_架构与合规设计说明.md) 为准。

## 二、整体架构与主链路

### 2.1 主业务链路

```
订单(已支付) → 运营确认并分配任务 → 任务(5段) → 报告管理(用印/下发/确认或申诉/归档) → 任务终态 → 订单终态
```

- **任务 5 段**：0 配置 → 1 采集（含待审核）→ 2 计算（LCA）→ 3 核查 → 4 报告（仅跳转报告管理）。定义见 [02_全局数据字典与枚举](../01_业务与流程/02_全局数据字典与枚举.md) §1.2、[04_任务调度与状态机](../01_业务与流程/04_任务调度与状态机.md)。
- **报告阶段**：所有报告相关操作（用印版上传、勾选下发、供应商确认接收/申诉、申诉处理、确认归档并上链）**仅在报告管理**完成；任务列表阶段 4 仅展示「报告」并跳转报告管理。详见 [01_整体业务与产品设计审查](../01_业务与流程/01_整体业务与产品设计审查.md) §1.7、§1.8。

### 2.2 三端职责与数据流

| 端 | 职责 | 与主链路的关系 |
|----|------|----------------|
| 中心端（运营） | 订单确认与任务派发；任务配置/采集审核/LCA 调度/核查验收；报告管理（用印、下发、申诉处理、归档）；供应商与核查机构管理；模板中心；系统管理 | 驱动订单→任务→报告全流程；与供应商/核查端通过同一 taskId/orderNo 对齐 |
| 供应商端 | 待办列表（待填报/已驳回/待澄清）→ 填报或澄清 → 我的报告（预览、确认接收或申诉）；企业信息与证照 | 采集阶段提交数据与凭证；报告阶段确认接收或申诉；身份与证照供运营/核查端使用 |
| 核查机构端 | 待核查任务列表与详情；报告/证书记录；通过/驳回/澄清（与运营端协作日志衔接） | 对应任务阶段 3 核查；核查结果与报告管理衔接 |

数据与状态以 **taskId、orderNo** 为统一关联键；供应商端待办类型、报告可见性、核查端任务与 [02](../01_业务与流程/02_全局数据字典与枚举.md)、[04](../01_业务与流程/04_任务调度与状态机.md)、[06](../02_功能与对接/06_供应商工作台功能清单与信息结构.md)、[09](../02_功能与对接/09_核查机构端功能清单与入口说明.md) 一致。

## 三、中心端设计方案

### 3.1 模块与页面

| 模块 | 页面/入口 | 第一阶段设计要点 |
|------|-----------|------------------|
| 运营驾驶舱 | operator/dashboard.html | 简版：概览 KPI、任务与数据流、快捷入口；二迭代做整体 |
| 订单管理 | operator/order.html | 订单四态（UNPAID/PAID/CONFIRMED/COMPLETED/REVOKED）；对已支付订单执行「确认并分配任务」生成任务；与鑫采商城集成为实施期对接 |
| 任务管理 | self_operated_task_list.html、task_detail_*.html、task_workspace.html | 列表按 5 段展示；阶段 4 仅「报告」并跳转报告管理；配置→采集→计算→核查在详情/工作台完成；LCA 当前为手工摆渡，预留接口 |
| 报告管理 | report_mgt.html、report_detail.html | 用印版上传（版本记录）→ 勾选报告/证书下发 → 申诉处理（驳回/接受）→ 确认归档并上链；待归档/已归档不展示申诉按钮；申诉中不可归档 |
| 供应商管理 | supplier_mgt.html、supplier_mgt_detail | 供应商 CRM、空间身份与业务关系；与供应商端 identity 证照数据对应 |
| 核查机构管理 | certifier_mgt.html、certifier_detail | 核查机构维护、商业策略配置；与核查端任务对应 |
| 模板中心 | templates_mgt.html、template_detail.html | 采集模板维护；任务配置时选模板生成 Snapshot；预留易碳数科 LCA 对接 |
| 结算中心 | settlement.html | 本迭代不做，仅预留入口 |
| 系统管理 | system_mgt.html | RBAC（角色-权限）、操作日志；范围见 02 §3 |

### 3.2 状态与规则摘要

- **订单状态**：见 02 §1.1；订单 COMPLETED = 其下全部任务最终状态为 completed；已作废由任务/报告终态或业务规则判定。
- **任务状态**：CONFIG → COLLECT ↔ AUDIT（采集子状态）→ CALC → VERIFY → CERTIFIED；列表将 COLLECT/AUDIT 合并为「采集」；阶段 4 仅跳转报告管理。
- **报告主状态**：process / archived / revoked；申诉为叠加维度；申诉中不可归档；待归档/已归档不展示申诉处理。见 02 §1.3、04 §2.5–2.6。
- **系统管理**：角色-权限矩阵示例、关键操作日志字段见 [02 §3](../01_业务与流程/02_全局数据字典与枚举.md)；原型阶段可占位，开发阶段细化。

## 四、供应商端设计方案

### 4.1 功能与页面

| 功能 | 页面 | 第一阶段设计要点 |
|------|------|------------------|
| 工作台 | supplier/dashboard.html | 待办数量、待澄清数量、报告数；快捷入口：待办任务、我的报告、返回门户 |
| 待办任务 | supplier/task_list.html | 列表字段：任务编号、订单号、产品、规格、**类型/状态**、最新通知。**类型**：待填报(pending_fill)、采集已驳回(rejected_collect)、待澄清/待整改(clarification)。**操作**：「填报」→ task_fill.html?taskId=；「处理」→ task_clarify.html?taskId=。与 06 表 2.2、Mock 一致 |
| 任务填报 | supplier/task_fill.html | 产品与订单只读；工序数据填报与凭证上传；采集已驳回时展示驳回原因；提交后运营端可审核通过/驳回 |
| 待澄清处理 | supplier/task_clarify.html | 任务与产品只读；澄清/驳回说明展示；回复说明与佐证上传；提交回复与运营端协作日志衔接 |
| 我的报告 | supplier/reports.html | 报告**仅在运营下发后**出现；每条：预览、确认接收、对报告有异议（申诉）；**确认接收后不可再申诉**。与 report_mgt 下发、07 推荐演示路径一致 |
| 企业信息与证照 | supplier/identity.html | 企业基本信息、联系人、证照（营业执照必选，ISO 等可选）；「提交至空间」/「从空间同步」；优先支持应用内维护路径 B，见 06 §2.4 |
| 知识库与帮助 | 占位或 help | 与供应商端帮助规划一致 |

### 4.2 数据实体与 Mock

- **待办任务**：taskId, orderNo, productName, specs, taskType, rejectReason?, clarificationText?；taskType 与运营端 COLLECT/AUDIT 及转派澄清对应。
- **报告**：报告仅在运营下发后出现在供应商端；received 标记确认接收后不可申诉。与 js/mockOperator.js、report_mgt 下发记录一致，taskNo = taskId。
- **推荐演示路径**：使用 [07_Mock数据说明](../02_功能与对接/07_Mock数据说明.md) 7.2 中一致订单号/任务号，保证运营下发后供应商「我的报告」可见、确认/申诉闭环可演示。

## 五、核查机构端设计方案

### 5.1 功能与页面

| 功能 | 页面 | 第一阶段设计要点 |
|------|------|------------------|
| 核查仪表盘 | certifier/dashboard.html | 简版：欢迎区、KPI 卡片（即将超时/新回复/待办总数/本周完成）、高优先级任务队列等 |
| 核查任务列表 | certifier/task_list.html | 待核查任务列表；taskId/orderNo 与运营端一致；操作「核查」跳转详情 |
| 核查任务详情 | certifier/task_detail.html | 任务与产品信息；供应商证照（只读）；核算报告与凭证（只读）；通过/驳回/发起澄清。**穿透**可为简化版（只读展示），不强制完整穿透 |
| 证书颁发管理 | certifier/certificates.html | 证书记录列表、搜索、查看；与运营端报告管理、任务阶段 3 衔接 |

第一阶段不强制「完整穿透查看」，以任务列表 + 任务详情 + 证书记录为准；穿透可只读展示，与 09、符合性检查结论一致。

### 5.2 与运营端/供应商端关系

- 待核查任务对应运营端任务阶段 3（VERIFY）；完成通过/驳回/澄清后，运营端任务状态与报告管理衔接。
- 供应商证照数据来源：可信数据空间或运营端已同步数据，按 taskId → 供应商 → 证照拉取；与 06 identity、中心端供应商详情一致。
- Mock 与 07、运营端 MOCK_TASK_MAP 保持一致；演示时同一 taskId 可走通运营↔核查闭环。

## 八、验收标准（设计侧）

与 [设计落地执行方案](设计落地执行方案.md) §2.3 一致，3 月 27 日 MVP 设计侧验收建议逐项确认：

- **文档**：01/02/03/04 与 06/09 中订单—任务—报告、报告下发/确认/申诉/归档规则一致；02 §3 系统管理范围明确；11 首段明确中心连接器本阶段仅储备、不生产。

- **Mock/联调**：07 推荐演示路径中订单号/任务号在三端一致，闭环可演示。
