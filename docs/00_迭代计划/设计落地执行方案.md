# 设计落地执行方案

本方案依据 [迭代计划与排期说明](迭代计划与排期说明.md) 与 [第一阶段设计符合性检查](第一阶段设计符合性检查.md)，将迭代计划与符合性结论转化为**可执行的设计与验收动作**，按阶段、按类型落地。

---

## 一、总体思路

- **第一阶段（当前至 3 月 27 日）**：补齐设计文档中已识别的缺口（RBAC/日志、中心连接器阶段说明已做）、验证原型与 Mock 能否支撑「待办→填报/澄清→报告→申诉」闭环及三端演示，输出 MVP 验收清单。
- **第二阶段（3 月后）**：设计侧预留接口与任务类型扩展点，文档中明确委托-SaaS 的边界与对接方职责。
- **第三阶段（4 月后）**：设计侧预留轻量化与混合流程的边界，文档中明确边端协议与编排职责。

以下执行项均标注**产出物**与**验收标准**，便于排期与对单。

### 2.3 3 月 27 日 MVP 验收清单（设计侧）

设计文档与原型联合验收，建议逐项打勾：

- [ ] **文档**：01/02/03/04 与 06/09 中订单—任务—报告、报告下发/确认/申诉/归档规则一致，无冲突。
- [ ] **文档**：02 §3 系统管理（RBAC/日志）范围明确；11 首段明确第一阶段中心连接器仅储备、不生产。
- [ ] **原型**：运营端可走通 订单→确认并分配任务→任务 5 段（配置→采集→计算→核查→报告）→报告管理（用印版、下发、申诉处理、归档）。
- [ ] **原型**：供应商端可走通 待办列表（类型：待填报/已驳回/待澄清）→填报/澄清→我的报告（预览、确认接收或申诉）。
- [ ] **原型**：核查端可展示 核查仪表盘、任务列表、报告/证书记录；与运营端同一 taskId 可对应。
- [ ] **Mock/联调**：07 推荐演示路径中订单号/任务号在运营、供应商、核查端一致，闭环可演示。

## 五、执行顺序与责任建议

1. **立即（第一阶段收尾）**：完成 D1–D4 复核与必要补写；按 P1–P6 逐项校验原型与 Mock，修复与 06/09/07 不一致处；用 2.3 节清单做 3 月 27 日 MVP 设计侧验收。
2. **3 月（与第二阶段前期并行）**：完成 D5–D7，保证技术方案与接口讨论有文档锚点。
3. **4 月前**：完成 D8–D9，第三阶段方案评审时有设计依据。

设计文档更新由产品/架构主导，原型与 Mock 校验由前端/全栈与产品共同执行；验收清单建议由项目经理或产品在 3 月 27 日前组织走查并签字。
