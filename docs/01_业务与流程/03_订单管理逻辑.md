# 订单管理逻辑

## 1. 订单状态流转（与 02 全局数据字典一致）

订单状态为四态，驱动条件如下：

| 状态 | 说明 | 触发条件 |
|------|------|----------|
| **UNPAID** 待支付 | 订单创建后的初始状态 | 商城下单生成订单 |
| **PAID** 已支付 | 支付成功，尚未进入生产 | 支付回调成功 |
| **CONFIRMED** 已确认 | 运营已确认，可生成并执行子任务 | 运营**手动**确认；确认时需选择**任务类型**（自营/委托）及**委托方**（若为委托任务，如擎工互联、易碳数科等） |
| **COMPLETED** 已完成 | 订单下所有子任务均已结束 | 所有关联子任务均到达终态（如已发证/已归档） |

**流转简述**：订单自商城下单后处于待支付（UNPAID）；**支付**成功即进入已支付（PAID）。运营在订单列表中对该订单执行**确认**并选择任务类型与委托方后，订单变为已确认（CONFIRMED），系统可据此**分配/生成子任务**。确认后生成的子任务可在运营端「任务管理」列表（自营任务/委托任务）中查看并处理，与订单详情中的「关联任务」一一对应。子任务经配置、采集、计算、核查、发证至归档全流程结束后，当**所有子任务均完成**时，订单自动或由系统汇总更新为已完成（COMPLETED）。

## 2. 订单与任务的关系

* 一个订单可关联多个**子任务**（按产品/SKU 或核算单元拆分）。
* 订单 **COMPLETED** 由子任务完成情况汇总决定：当所有子任务均结束（如状态为已发证、已归档等）时，订单可置为已完成。
* 报告管理/归档针对的是任务发证后的产出；任务进入「待归档」「已归档」后，在报告管理页进行查看与归档操作，与订单状态解耦，但订单完成依赖这些任务终态。
* **订单与子任务状态映射**：订单 COMPLETED 当且仅当其下全部子任务均到达终态（如 CERTIFIED 已发证、或运营侧「已归档」）。报告管理页的「待归档」「已归档」与任务状态对应；归档触发条件：任务已发证（CERTIFIED）后，由运营在报告管理页执行归档操作，归档后任务为「已归档」。
* **发证后至归档的链路**（详见 04 任务调度与状态机）：任务发证后，运营将报告/证书**下发**至供应商；**供应商确认接收**后，运营方可执行待归档/归档，任务阶段进入「待归档」「已归档」。未确认接收前任务处于「已下发」阶段。

## 3. 运营端订单列表与筛选

* 订单列表展示字段包含：订单编号、核算产品、购买企业、服务周期、任务进度、**订单状态**等。
* 筛选条件包含：订单编号、企业名称、**订单状态**（待支付/已支付/已确认/已完成），与上述四态一致。

## 4. 运营端菜单与委托方说明

* **委托业务与多委托方**：委托任务存在多个委托方（如擎工互联、易碳数科等）。商城侧不指定委托方，由运营在**确认订单**时选择任务类型（自营/委托）及委托方。
* **菜单层级**：运营端仅做「任务管理 → 自营任务 | 委托任务」两级；不把各委托方做成三级菜单，避免菜单过长。委托方在**委托任务**页内通过「委托方」筛选器选择即可。
* **委托任务页**：占位页 `entrusted_task_list.html` 提供委托方下拉筛选与列表，后续对接真实接口。
