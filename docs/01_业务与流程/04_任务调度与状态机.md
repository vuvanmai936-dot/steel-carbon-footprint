# 任务调度与状态机

## 1. 页面形态说明
* **多页任务详情**：任务列表点击「处理」等入口，进入按阶段拆分的多页详情（如 `task_detail_config.html` / `task_detail_collect.html` / `task_detail_lca.html` / `task_detail_verify.html`），每页对应一个主阶段，便于深度操作与权限隔离。
* **可选 SPA 工作台**：`task_workspace.html` 为**单页应用 (SPA)** 形态，基于 `activeStep` 变量在同一页内切换配置 / 采集与审核 / 计算 / 核查 / 发证 等视图，适用于一站式轻量操作或演示。与多页详情二选一或并存，由产品场景决定。

## 2. 阶段详细逻辑

### Stage 1: 配置 (Config)
* **前置条件**: 任务状态为 `CONFIG`。
* **数据来源**: 调用 `模板中心` 接口，拉取状态为 `Active` 的模板列表。
* **动作**: 运营人员选择模板后，系统需将模板结构（Sheet/Items）复制一份到当前任务实例中（Snapshot），避免后续模板变更影响进行中的任务。

### Stage 2: 采集与审核 (Collect & Audit)
* **状态说明**：`COLLECT`（待填报）与 `AUDIT`（待审核）在任务列表/看板中**合并为「采集」阶段展示**；**待审核 (AUDIT) 为采集阶段子状态，任务列表不单独列出待审核阶段**。详情页内可区分「待填报」「待审核」子状态（参见 02 全局数据字典）。
* **视图逻辑**:
  * 左侧展示 Excel 结构（Tabs 分组）。
  * 右侧展示对应工序的“凭证要求”。
* **审核动作**:
  * **驳回**: 需弹窗填写驳回原因，任务状态回退至 `COLLECT`（供应商需重填）；供应商在待办中可见该任务，进入填报页可查看驳回原因。
  * **通过**: 任务状态流转至 `CALC`。
* **供应商端待澄清/待整改**：当运营在 LCA 或核查阶段将澄清「转派给供应商」时，该任务在供应商侧视为「待澄清/待整改」；供应商需在限定时间内补充数据、上传新报告或回复澄清，提交后等待运营/认证机构确认。

### Stage 2.5: LCA 计算 (CALC)
* **CALC 即 LCA 计算**；LCA 为**独立系统**（当前部署在南钢），用于产品碳足迹核算。
* **当前**：运营在 LCA 系统中手工创建模型、**手工摆渡**完成填写与计算（在 LCA 系统页面操作）。
* **后续迭代**：预留与南钢 LCA 的接口对接（如 3.6.1 模型产品列表、3.6.2 提交产品数据、3.6.3 获取计算结果、3.6.4 获取 LCIA 结果等）；MVP 阶段不做接口开发。与 TDS/NDI 的接口与互操作要求见 **02 功能与对接 → [11_接口与互操作设计（实施期）](../02_功能与对接/11_接口与互操作设计（实施期）.md)**。

### Stage 3: 三方核查 (Interface 4.4)
* **交互逻辑**: 
  * 这是一个**异步等待**过程。
  * 右侧“协作日志”需实时轮询（或 WebSocket）三方系统的消息；**原型阶段**协作日志为静态/模拟数据，正式实现需接入轮询或 WebSocket。
  * 若收到 SGS 的“澄清请求”，需**高亮显示并在日志中弹窗提醒**（此为必选交互规范）。
* **核查完成**：当**核查机构已上传核查报告与核查声明证书**后，该任务的「核查任务」在任务管理侧即告完成；后续与报告/证书相关的所有操作（用印版上传、下发、供应商确认与申诉、待归档、归档上链）**仅在报告管理中处理**（参见 2.4 任务管理与报告管理的边界）。

## 2.4 任务管理与报告管理的边界（已采纳，控全局）
* **任务阶段（5 段）**：**0 配置、1 采集、2 计算、3 核查、4 报告**。任务管理仅负责 0–3（核查任务）；**阶段 4「报告」**统一表示「与报告/证书相关的后续全部在报告管理中处理」。
* **任务列表**：阶段 4 仅展示为**「报告」**（可带子状态文案如待处理/已下发/已归档/已作废，仅展示用）；点击后**统一跳转报告管理**，不在任务详情中提供报告下发、用印版上传或归档入口。
* **报告管理**：自「三件套齐备」起负责：上传用印版 → 勾选报告/证书下发 → 供应商确认接收或申诉 → 待归档 / 确认归档并上链。**任务最终状态**由报告侧结果驱动：归档完成 → 任务「已完成」；报告/任务作废或终止 → 任务「已作废」→ 用于驱动订单（见 2.7）。
* **发证**：业务上的「发证」（如 VC 上链、赋码）可在任务侧保留为状态节点或轻量确认；报告/证书对供应商的下发、版本与归档统一在报告管理完成。

## 2.5 报告归档触发条件与下发流程
* **下发前**：运营在**报告管理 → 档案详情**内上传线下用印完成的核算报告（须先收到核查报告与核查声明证书），系统以**版本记录**（草稿 → 用印版 V1）作为下发前置条件。
* **下发**：运营在报告管理页通过**复选框**勾选要下发的报告/证书，确认后执行「下发」至供应商；供应商端可「确认接收」或「我有异议」申诉。
* **待归档/已归档**：**只有供应商确认接收后**，运营方在报告管理中执行「确认归档并上链」，报告才进入已归档（archived）。**订单 COMPLETED** 依赖其下所有任务**最终状态**均为「已完成」（参见 2.7、03 订单管理逻辑）。
* **申诉中不可归档**：存在待处理申诉（申诉中）时，**不得执行归档**；须先处理申诉（驳回或接受）或等待供应商确认接收后，方可执行「确认归档并上链」。实现上归档按钮在申诉中时应置灰并提示「需先处理申诉或等待供应商确认接收后再归档」。

## 2.6 供应商报告异议/申诉（事后流程）
* **申诉建立时机**：仅当**运营方已对该任务执行报告/证书下发**后，供应商在「我的报告」中可见该报告；供应商可**预览**后选择「确认接收」或「不接收并提起申诉」。
* **确认接收后**：供应商一旦「确认接收」，则**不可再申诉**，流程终止；此后运营方可执行待归档/归档，报告进入待归档、已归档。
* **待归档、已归档与申诉无关**：只有供应商确认接收后才会出现待归档、已归档，故这两态下**不会存在待处理申诉**，运营端**不展示「驳回申诉」或「接受申诉」**按钮。
* **申诉处理**：对「已下发、供应商未确认接收」且存在申诉的报告，运营可**驳回**（维持原报告）或**接受**（触发后续流程）。申诉通过后建议采用「原任务回退至 VERIFY（重新核查）」：同一 taskId 增加申诉轮次标识，重新走核查→发证→待归档；若业务要求仅重算不重核查，可回退至 CALC。可选扩展：「新建申诉处理任务」专门做重算/重核查。
* **原型阶段说明**：当前原型仅演示运营端「驳回申诉」「接受申诉」操作；接受申诉后业务上应触发「原任务回退至 VERIFY/CALC」或「新建申诉处理任务」，该部分为**后续迭代**，报告管理页不展示回退/新建任务入口，正式实现后可在任务列表中查看重核查任务。
* **申诉中不可归档**：申诉中（已下发、未确认接收、存在待处理申诉）时不允许归档；须先处理申诉或等待供应商确认接收后再执行归档（见 2.5）。
* **报告状态约定**：报告管理采用**主状态**（流转中 process / 已归档 archived / 已作废 revoked）表示归档进度；**申诉中**为叠加维度，仅出现在「已下发、供应商未确认接收」的报告上；待归档/已归档不展示申诉处理操作。

## 2.7 任务阶段（五段）与任务最终状态、订单对应（控全局）
* **任务列表采用 5 段**：**0 配置、1 采集、2 计算（LCA）、3 核查、4 报告**。stageIndex 4「报告」= 所有报告相关操作入口统一指向报告管理，列表可展示子状态（如待处理/已下发/已归档/已作废）仅作信息展示，不增加阶段数。
* **任务最终状态**（由报告侧结果驱动，用于控订单）：
  * **进行中**：阶段 0–3 或阶段 4 报告尚未终态。
  * **已完成**：报告已**归档完成**（运营在报告管理中执行「确认归档 & 上链」后）→ 对应订单可汇总为 **COMPLETED**。
  * **已作废**：任务或报告**被迫终止**（如报告/档案作废、异常关闭）→ 对应订单可判定 **已作废**。
* **与订单的对应**：**订单 COMPLETED** = 其下全部任务最终状态为「已完成」；**订单已作废** = 存在任务最终状态「已作废」或按业务规则判定。详见 03 订单管理逻辑。
* **原型/实现要点**：任务列表阶段 4 展示「报告」并跳转报告管理；报告管理已支持用印版上传、复选框下发、申诉处理、确认归档并上链。

## 2.8 上链触发时机（报告归档）
* **报告管理侧**：**仅在运营点击「确认归档 & 上链」时触发上链**。即档案详情页中，当报告状态为「流转中」、三件套齐备且满足归档条件时，运营执行「确认归档 & 上链」后，系统对交付物做哈希计算并上链存证，报告状态变为「已归档」；此前（下发、用印版上传、供应商确认接收等）均不上链。
* **任务/发证侧**：任务详情中的「已上链」指**发证阶段**的数字凭证（VC）上链，与报告管理中的「归档上链」为不同动作；报告档案的存证上链仅与归档操作绑定。

## 2.9 报告版本记录控制
* **核算报告版本**：当前为二态——**草稿**（系统生成/流转给核查机构）、**用印版 V1**（运营在详情页上传线下用印版后记录）。上传用印版即生成一条版本记录（草稿 → 用印版 V1），并作为下发前置条件。
* **流转记录 (history)**：所有关键操作均追加一条记录（用印版上传、核查报告/证书同步、下发至供应商、归档上链成功、档案作废等），仅追加、不删除，便于审计与追溯。
* **控制原则**：归档前可上传用印版并更新版本；**归档后版本与流转记录锁定**，不再变更。若业务后续支持多版用印版（V1、V2…），建议版本号仅递增、且历史记录只读。

## 2.10 数字合约与业务流程对应

关键业务节点与数字合约的对应关系如下（设计级对应；原型阶段不实现真实合约签署与链上存证，正式实施时需与数字合约的创建、签署、执行、存证环节一一映射）。依据《可信数据空间 数字合约技术要求》。

| 业务节点 / 阶段 | 数字合约作用 | 签署/执行/存证要点 |
|-----------------|-------------|-------------------|
| 订单确认 / 服务开通 | 数据使用与服务授权合约 | 运营确认并分配任务前后，由平台/空间发起签署或自动触发；存证与订单/任务创建关联 |
| 模板下发 / 任务配置 | 数据采集范围与使用目的合约 | 配置完成、Snapshot 生成时绑定合约条款；执行以「仅限本任务使用」等策略体现；存证与 taskId 关联 |
| 供应商填报提交 | 数据提交与真实性声明 | 提交时勾选或自动关联声明类合约；存证与提交事件、taskId 关联 |
| 报告下发 / 供应商确认接收 | 报告交付与接收确认合约 | 下发即合约生效、确认接收即合约完成；存证与 reportId、接收时间关联 |
| 归档并上链 | 档案存证与不可篡改 | 归档动作为合约执行结果上链；存证与档案版本、哈希关联 |

状态机与报告管理中的「确认并分配任务、配置完成、提交、下发、确认接收、确认归档并上链」等节点，在正式实施时需与上述数字合约环节可追溯对应。

## 3. 协作日志 (Log System)
日志需记录以下维度的操作：
* **时间**: 精确到秒。
* **操作人**: 系统 / 运营 / 供应商 / 核查员。
* **动作**: 状态变更、文件上传、澄清发起、供应商申诉。

**澄清、驳回与异议的参与方与可见性**：各端仅展示其有权参与的澄清会话与相关消息；运营端可见任务下全部澄清/驳回/异议，供应商与核查机构仅见归属自己的会话。核查阶段核算报告草稿类澄清仅运营↔核查机构可见；供应商数据类澄清须经运营确认后决定自行回复或转发给供应商。报告阶段异议（申诉）的消息与可见性同上述规则。详见 [18_驳回澄清与消息全局规划](../04_澄清与消息/18_驳回澄清与消息全局规划.md)。
