# 任务调度与状态机

## 1. 页面形态说明
* **多页任务详情**：任务列表点击「处理」等入口，进入按阶段拆分的多页详情（如 `task_detail_config.html` / `task_detail_collect.html` / `task_detail_lca.html` / `task_detail_verify.html`），每页对应一个主阶段，便于深度操作与权限隔离。
* **可选 SPA 工作台**：`task_workspace.html` 为**单页应用 (SPA)** 形态，基于 `activeStep` 变量在同一页内切换配置 / 采集与审核 / 计算 / 核查 / 发证 等视图，适用于一站式轻量操作或演示。与多页详情二选一或并存，由产品场景决定。

## 2. 阶段详细逻辑

### Stage 1: 配置 (Config)
* **前置条件**: 任务状态为 `CONFIG`。
* **数据来源**: 调用 `模板中心` 接口，拉取状态为 `Active` 的模板列表。
* **动作**: 运营人员选择模板后，系统需将模板结构（Sheet/Items）复制一份到当前任务实例中（Snapshot），避免后续模板变更影响进行中的任务。

### Stage 2: 采集与审核 (Collect & Audit)
* **状态说明**：`COLLECT`（待填报）与 `AUDIT`（待审核）在任务列表/看板中合并为「采集」阶段展示；详情页内可区分「待填报」「待审核」子状态（参见 00 全局数据字典）。
* **视图逻辑**:
  * 左侧展示 Excel 结构（Tabs 分组）。
  * 右侧展示对应工序的“凭证要求”。
* **审核动作**:
  * **驳回**: 需弹窗填写驳回原因，任务状态回退至 `COLLECT`（供应商需重填）。
  * **通过**: 任务状态流转至 `CALC`。

### Stage 3: 三方核查 (Interface 4.4)
* **交互逻辑**: 
  * 这是一个**异步等待**过程。
  * 右侧“协作日志”需实时轮询（或 WebSocket）三方系统的消息；**原型阶段**协作日志为静态/模拟数据，正式实现需接入轮询或 WebSocket。
  * 若收到 SGS 的“澄清请求”，需**高亮显示并在日志中弹窗提醒**（此为必选交互规范）。

## 3. 协作日志 (Log System)
日志需记录以下维度的操作：
* **时间**: 精确到秒。
* **操作人**: 系统 / 运营 / 供应商 / 核查员。
* **动作**: 状态变更、文件上传、澄清发起。
