# 全局数据字典与枚举定义

## 1. 核心状态机 (State Machine)

### 1.1 订单状态 (Order Status)
| 状态值 | 描述 | 触发条件 |
| :--- | :--- | :--- |
| `UNPAID` | 待支付 | 订单创建初始状态 |
| `PAID` | 已支付 | 支付回调成功 |
| `CONFIRMED` | 已确认 | 运营人员介入或系统自动确认 |
| `COMPLETED` | 已完成 | 所有关联子任务均结束 |
| `REVOKED` | 已作废 | 存在任务最终状态 `revoked` 或按业务规则判定作废 |

* 运营端订单列表支持按「已作废」筛选与展示；订单 **REVOKED** 由子任务/报告作废或业务规则派生，与任务最终状态对应。

### 1.2 任务状态 (Task Status) - **核心**
此状态流转应用于任务详情与工作台页面。
1. **`CONFIG` (待配置)**: 任务生成，等待运营人员选定模板。
2. **`COLLECT` (待填报)**: 模板下发，等待供应商提交数据；供应商提交后进入本阶段子状态 **`AUDIT` (待审核)**，运营人员检查数据/凭证完整性。
   * **列表/看板展示**：COLLECT 与 AUDIT 合并为「采集」阶段展示；详情页内可区分「待填报」与「待审核」。
   * **驳回**：回退至 COLLECT（待填报）；供应商在待办中仍见该任务，进入填报页可查看驳回原因并重填；**通过**：流转至 CALC。
3. **`CALC` (LCA 计算)**：数据无误后进入本阶段。**LCA 为独立系统**（当前部署在南钢），用于产品碳足迹核算。**当前**由鑫智链/产研院运营人员在 LCA 系统中**手工创建模型、手工摆渡**（在 LCA 系统页面填写并计算）；**后续迭代**预留与南钢 LCA 系统的**接口对接**（如模型列表、提交产品数据、获取计算结果/LCIA 结果等），参见接口文档 3.6。
4. **`VERIFY` (核查中)**: 报告生成，推送到 SGS/DNV 系统。
5. **`CERTIFIED` (已发证)**: 核查通过，生成 VC 证书并上链。发证后与报告相关的所有操作（用印版上传、下发、供应商确认/申诉、待归档、归档上链）**仅在报告管理中处理**；任务列表仅保留**阶段 4「报告」**，点击跳转报告管理（参见 01 文档 1.8、04 文档 2.4）。

**任务列表阶段（5 段）**：**0 配置、1 采集、2 计算、3 核查、4 报告**（stageIndex 0–4）。阶段 4 可展示子状态文案（如待处理/已下发/已归档/已作废）仅作展示，不增加阶段数。

**任务最终状态（控订单）**：

| 最终状态值 | 描述 | 与订单的对应 |
| :--- | :--- | :--- |
| `in_progress` | 进行中 | 订单仍为 CONFIRMED 等非终态 |
| `completed` | 已完成 | 报告已归档完成 → 订单可汇总为 **COMPLETED** |
| `revoked` | 已作废 | 任务/报告被迫终止 → 订单可判定 **已作废** |

* 订单 **COMPLETED**：其下全部任务最终状态均为 `completed`；订单 **已作废**：存在任务 `revoked` 或按业务规则判定。详见 03 订单管理逻辑、04 文档 2.7。

### 1.3 报告主状态 (Report Status) — 报告管理页
报告/档案包在运营侧使用**主状态**表示归档进度（互斥三选一），与任务阶段对应关系见 04 任务调度与状态机。

| 状态值 | 描述 | 说明 |
| :--- | :--- | :--- |
| `process` | 流转中 | 已发证、未在运营侧归档；满足归档条件且**非申诉中**时可执行「确认归档并上链」 |
| `archived` | 已归档 | 已归档上链，可下载、可撤销归档 |
| `revoked` | 已作废 | 人工撤销归档，仅可查看/追溯 |

* **申诉中 (APPEAL_PENDING)**：为**叠加字段**，表示该报告存在待处理申诉。申诉仅在**运营下发报告后、供应商确认接收前**由供应商发起。**申诉中不可归档**：存在待处理申诉时不得执行「确认归档并上链」，须先处理申诉或等待供应商确认接收（见 04 文档 2.5、2.6）。**待归档、已归档**仅在供应商确认接收后出现，故该两态下不会存在待处理申诉，运营端不展示申诉处理按钮。处理完成后移除申诉标记，主状态按业务规则是否变更见 04 文档 2.6 节。

#### 1.3.1 报告主状态设计说明与合理性

报告主状态采用**三态互斥 + 申诉叠加**的设计，用于表示归档进度并与任务/订单终态对齐。

**设计要点**

- **三态互斥**：`process`（流转中）、`archived`（已归档）、`revoked`（已作废）覆盖「进行中」「正常终态」「异常终态」，互斥不重叠，便于列表筛选、权限与统计。
- **与业务流程一致**：报告管理负责「用印版上传 → 下发 → 供应商确认/申诉 → 确认归档并上链」；「是否已归档」是核心节点，用 process / archived 区分即可；revoked 与任务最终状态「已作废」、订单「已作废」一致，便于驱动订单与任务汇总。
- **申诉作为叠加**：同一份报告在未归档前可能经历「无申诉 → 有申诉 → 处理完申诉」，若将「申诉中」单独设为主状态会与「已下发待确认」等交叉且需定义申诉关闭后的回退状态。将申诉作为叠加维度，并约定「申诉中不可归档」，逻辑简单、状态机清晰。
- **终态明确**：archived 与 revoked 为两个终态，对应任务/订单的「已完成」与「已作废」，便于汇总与报表。

**可选细化（不改变主状态）**

- **「流转中」内涵较宽**：process 覆盖未下发、已下发待确认、待归档、申诉中等。若运营端需要在报告管理页按「待下发 / 已下发待确认 / 待归档」等细分筛选，建议不增加主状态，而在报告管理增加**子状态或筛选维度**（如按是否已下发、是否待归档），与任务列表阶段 4 的 reportSubStatus 思路一致。
- **撤销归档**：已归档可「撤销归档」后变为 revoked；revoked 仅可查看/追溯。若业务要求「撤销后不允许再次归档」，当前设计即可；若将来需支持「撤销后仍可再次归档」，再考虑扩展子状态或状态值。
- **与任务阶段 4 对应**：任务侧子状态为待处理 / 已下发 / 待归档 / 已归档 / 已作废；报告主状态 process 覆盖未归档前的多种情况，archived = 已归档，revoked = 已作废，对应关系清晰，无需调整主状态。

**结论**：报告主状态三态 + 申诉叠加的设计可支撑归档进度表达、下发/确认/申诉规则及与任务/订单的一致性；若需更细的运营侧筛选，建议通过子状态或筛选条件扩展，主状态保持当前设计。

### 1.4 报告相关术语与各端对照

| 概念 | 运营端报告管理 | 任务列表阶段 4 | 供应商「我的报告」 |
|------|----------------|----------------|---------------------|
| 流转中 | 主状态 process：已发证、未归档，可下发/归档（非申诉中） | 子状态「待处理」等 | 已下发未确认时可见 |
| 已下发 | 已执行「下发」，供应商可见 | 子状态「已下发」 | 可见，可确认接收或申诉 |
| 待归档 | 供应商已确认接收，待运营执行「确认归档并上链」 | 子状态「待归档」 | 确认接收后不可申诉 |
| 已归档 | 主状态 archived，已上链 | 子状态「已归档」 | — |
| 档案 | 报告管理中的交付物（核算报告+核查报告+证书等） | 同任务产出，跳报告管理查看 | 报告 |

任务阶段 4「报告」= 所有报告相关操作入口统一在报告管理；任务列表仅展示阶段 4 及子状态并跳转报告管理。

## 2. 角色权限 (Roles)
* **L1 商城用户**: 下单、支付、签署商务协议。
* **L2 供应商 (Supplier)**: 在供应商工作台填报数据、上传凭证。供应商端待办类型：待填报、采集已驳回（需重填并可查看驳回原因）、待澄清/待整改（来自 LCA/核查阶段转派，需补充数据或回复澄清）。
* **L3 运营人员 (Operator)**: 配置模板、审核数据、调度任务。
* **L4 核查机构 (Verifier)**: 查看穿透数据、反馈核查意见。
