# 系统管理：RBAC 与审计日志范围（第一阶段 MVP）

本文档界定**正式实施阶段**运营端系统管理中的 **RBAC（基于角色的访问控制）** 与 **审计日志** 的产品范围与字段约定，与 [00_迭代计划/第一阶段设计符合性检查](../00_迭代计划/第一阶段设计符合性检查.md)、[01_业务与流程/02_全局数据字典与枚举](../01_业务与流程/02_全局数据字典与枚举.md) 第 2 节角色说明、[05_标准与合规/22_TDS_架构与合规设计说明](../05_标准与合规/22_TDS_架构与合规设计说明.md) 第 2 节身份与审计上报配套使用。  
**原型阶段**：[`operator/system_mgt.html`](../../operator/system_mgt.html) 为界面占位；权限与日志以本文档与后端实现为准。

---

## 1. 目标与原则

- **目标**：满足排期「系统管理：RBAC 权限与日志」的 MVP 要求；支撑「谁能进哪些菜单、谁能做哪些关键动作」与「关键动作可追溯」。
- **身份源**：正式环境以可信数据空间/行业功能节点统一身份登记与鉴权结果为准（见 22 文档）；本应用消费 **用户标识、主体（企业）标识、角色、权限声明**，不重复建设主身份源。
- **最小够用**：第一阶段以**中心运营端** RBAC 与审计为主；供应商端、认证机构端以**租户内角色**（企业管理员/填报员、核查机构账号）为扩展，可与连接器侧策略同步，接口细节见 [11_接口与互操作设计（实施期）](11_接口与互操作设计（实施期）.md)。

## 2. RBAC 模型

### 2.1 实体

| 实体 | 说明 |
|------|------|
| 用户 | 与空间侧账号绑定；可属于一个或多个主体（企业）上下文（运营侧用户可视为平台主体）。 |
| 角色 | 命名集合，如「运营管理员」「运营操作员」「只读审计」。 |
| 权限点 | 细粒度 **资源 + 操作**，如 `report:issue`（报告下发）、`report:archive`（确认归档并上链）。 |
| 用户-角色 | 多对多；可按组织维度分配。 |
| 角色-权限 | 多对多；运营管理员预置「全量业务权限 + 系统管理」 |

### 2.2 与数据可见性

- **运营端列表/详情**：默认按平台数据权限过滤（如仅本部门任务）；是否按组织/条线隔离由实施期数据权限策略补充。
- **供应商 / 认证机构**：仅能访问与本企业/本机构相关的任务、报告、澄清会话（与 [18_驳回澄清与消息全局规划](../04_澄清与消息/18_驳回澄清与消息全局规划.md) 一致）。

## 3. 角色建议（第一阶段）

以下为产品级建议，实施时可增减；命名可与 IAM 统一。

| 角色 | 典型职责 | 系统管理 |
|------|----------|----------|
| 运营管理员 | 全业务模块、模板与伙伴管理、高风险操作（归档、作废、申诉终裁） | 可维护角色/用户/权限、查看全部审计日志 |
| 运营操作员 | 日常订单确认、任务配置与调度、采集审核、报告管理常规操作 | 无或仅查看系统管理 |
| 只读审计 | 列表/详情只读，不可触发变更类操作 | 仅查看审计日志（可选导出） |
| 供应商企业管理员 | 本企业成员与授权（若 MVP 不拆，可与填报员合并） | 不适用运营端 |
| 供应商填报员 | 待办填报、凭证、澄清、我的报告确认/申诉 | 不适用运营端 |
| 核查机构账号 | 核查任务处理、上传核查报告与声明证书、核查阶段澄清 | 不适用运营端 |

## 4. 权限点清单（运营端 · 建议）

按模块列出 **建议权限点**（开发阶段可映射为菜单 + API  scope）。「查看」与「操作」宜拆分，便于只读审计角色。

| 模块 | 建议权限点（资源:操作） | 说明 |
|------|-------------------------|------|
| 驾驶舱 | `dashboard:view` | 查看指标与入口 |
| 订单 | `order:view`、`order:confirm`、`order:assign_task` | 确认、分配任务与 03 订单逻辑一致 |
| 任务 | `task:view`、`task:config`、`task:audit_collect`、`task:lca_mark`、`task:verify_coord` | 配置、采集审核、LCA 阶段标记/摆渡说明、核查协作 |
| 报告管理 | `report:view`、`report:upload_sealed`、`report:issue`、`report:appeal_handle`、`report:archive`、`report:revoke` | 用印版、下发、申诉处理、归档上链、作废 |
| 供应商管理 | `supplier:view`、`supplier:edit` | 主数据维护 |
| 认证机构管理 | `certifier:view`、`certifier:edit` | 主数据维护 |
| 模板中心 | `template:view`、`template:edit`、`template:publish` | 模板发布与 Snapshot 源 |
| 结算中心 | `settlement:view` | 第一阶段可仅查看占位 |
| 系统管理 | `system:role_manage`、`system:user_manage`、`system:audit_view`、`system:config_view` | RBAC 与日志；配置类后续扩展 |

**高风险操作**（建议强制二次确认或双人复核，实施期定稿）：`report:archive`、`report:revoke`、`order:assign_task`（若对接生产订单）。

## 5. 审计日志范围

### 5.1 必须记录的事件（第一阶段）

下列事件**建议全量记录**（成功与失败均记录，失败带错误码/原因摘要）：

- **身份与会话**：登录成功/失败、登出（若由本应用感知）。
- **订单**：确认、分配任务、状态变更触发。
- **任务**：模板绑定/Snapshot 生成、采集提交（供应商侧可在供应商审计中记或同步）、采集审核通过/驳回、阶段状态变更（含 LCA/核查关键节点的人工标记）。
- **报告**：用印版上传、版本变更、勾选下发、供应商确认接收、申诉建立与处理（驳回/接受）、确认归档并上链、档案作废。
- **主数据**：供应商/认证机构/模板关键字段变更（谁改了什么）。
- **系统管理**：角色/权限/用户绑定变更。

供应商端、认证机构端对应操作（提交数据、上传核查报告等）建议在**各租户审计表**或统一审计服务中记录，字段与下文一致，便于跨端追溯。

### 5.2 日志字段（最小集）

| 字段 | 说明 |
|------|------|
| `id` | 日志唯一 ID |
| `occurredAt` | 操作时间（ISO8601，精确到秒或毫秒） |
| `actorUserId` / `actorName` | 操作人标识与展示名 |
| `actorRole` | 当时角色快照（可选，便于事后解释） |
| `action` | 枚举，如 `LOGIN`、`ORDER_CONFIRM`、`TASK_AUDIT_REJECT`、`REPORT_ISSUE`、`REPORT_ARCHIVE` |
| `resourceType` | `order` / `task` / `report` / `supplier` / `template` / `system` 等 |
| `resourceId` | 业务主键，如 orderNo、taskId、reportId |
| `outcome` | `success` / `failure` |
| `detail` | JSON 或文本摘要：请求参数关键字段、驳回原因代码、IP/UA（若合规允许） |
| `correlationId` | 可选，分布式追踪与报表 |

正式环境需与 **TDS/连接器审计上报** 对齐时，在 `detail` 或扩展字段中预留 `spaceAuditPayload` 映射（见 22 文档 1.3、2.3）。

### 5.3 保留与检索

- **保留期**：建议不少于 **180 天**，金融/监管若有更长要求则按监管；归档可导出为不可篡改存储（对象存储 WORM 等）。
- **检索**：按时间、操作人、`action`、`resourceId` 组合筛选；运营管理员与只读审计角色可访问（权限见 §4）。

### 5.4 与协作日志、消息的区别

- **协作日志**（任务详情右侧）：面向用户的业务时间线，偏展示与即时沟通上下文。
- **审计日志**（系统管理）：面向合规与追责，**只追加、不随业务 UI 删除**；二者可通过对 `taskId`/`reportId` 关联，但存储与权限模型分离。
