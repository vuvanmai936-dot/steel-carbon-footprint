# 任务详情页问题分析（从配置模板开始逐个调整）

本文档用于从「规则配置」页开始，逐个统一各任务详情页的**头部布局**与**信息条（task-info-dashboard）**，并列出当前各页的差异与问题。

---

## 一、页面范围

| 页面 | 文件 | 说明 |
|------|------|------|
| 规则配置 | task_detail_config.html | 主流程第 1 步，作为**模板基准** |
| 数据采集 | task_detail_collect.html | 主流程第 2 步 |
| LCA 计算 | task_detail_lca.html | 主流程第 3 步 |
| 核查认证 | task_detail_verify.html | 主流程第 4 步 |
| 报告引导 | task_detail_certify.html | 已弱化为「前往报告管理」引导页，无完整信息条 |

---

## 二、头部（el-header）问题

### 2.1 当前结构（多数页面）

- **左侧**：返回按钮 + 面包屑（中心运营端 / 任务管理 / 当前阶段名）
- **右侧**：SLA 胶囊 + 头像 + 用户名

### 2.2 不一致之处

| 问题 | 说明 | 涉及页面 |
|------|------|----------|
| **SLA 缺失** | 报告引导页（certify）右侧没有 SLA 胶囊，只有头像和用户名 | task_detail_certify.html |
| **SLA 文案不统一** | config 有「SLA 剩余时间 (截止: 2026-02-10)」，其余为「SLA 剩余时间」；数值各页写死且不同 | config / collect / lca / verify |
| **头部内联样式重复** | 各页 el-header 的 style 相同但分散写，不利于统一修改 | 全部 |

**建议**：以 config 为基准，约定「头部结构 + 右侧区域（SLA + 头像 + 用户名）」统一；certify 若保留为引导页，可保留无 SLA 或改为占位文案（如「—」）。SLA 数值后续可改为由 Mock/接口统一提供。

---

## 三、面包屑（Breadcrumb）

- 前两级一致：`中心运营端` / `任务管理`。
- 第三级为当前阶段名，各页不同，属合理差异：
  - config: 配置规则
  - collect: 数据采集
  - lca: LCA计算
  - verify: 核查验收
  - certify: 报告（已统一至报告管理）

**结论**：仅需统一「阶段名」用词规范（如全用中文、是否带括号），无结构性错误。

---

## 四、任务信息条（task-info-dashboard）问题

### 4.1 第一块：产品与任务

| 页面 | 规格展示 | 订单展示 | 问题 |
|------|----------|----------|------|
| config | `规格: {{ taskInfo.specs }}` | `<el-icon><Tickets /></el-icon>` + `订单: ` + 订单链接 | 基准 |
| collect | 同 config | 同 config | 与 config 一致 |
| lca | 仅 `{{ taskInfo.specs }}`，无「规格:」前缀 | 仅订单链接，无图标、无「订单:」文案 | 与 config 不一致 |
| verify | 仅 `{{ taskInfo.specs }}`，无「规格:」前缀 | 仅订单链接，无图标、无「订单:」文案 | 与 config 不一致 |

**建议**：lca、verify 与 config/collect 对齐，统一为「规格: xxx | 订单: 链接」，并保留订单图标（Tickets）。

### 4.2 第二块：供应商

| 页面 | 标题 | 供应商名称 | 副文案（等级/信用） | 问题 |
|------|------|------------|----------------------|------|
| config | 目标供应商 | el-link 可点（viewSupplier） | Level A + 信用良好 | 基准之一 |
| collect | 来源供应商 | el-link 可点 | Level A + 信用良好 | 标题与 config 用词不同 |
| lca | 目标供应商 | el-link 可点 | Level A 信用良好（一行） | 标题与 collect/verify 不同，副文案格式不同 |
| verify | 来源供应商 | **纯 span，不可点** | 仅 Level A | 缺少跳转、副文案不统一 |

**建议**：
- 标题统一：建议全部改为「**来源供应商**」（与业务含义一致：数据来自该供应商）。
- 供应商名称：全部使用 `el-link @click="viewSupplier"`，与 config/collect 一致。
- 副文案：统一为「Level A」+「信用良好」（或一行「Level A 信用良好」），verify 补全「信用良好」。

### 4.3 第三块：状态

| 页面 | 标题 | 说明 |
|------|------|------|
| config | 配置状态 | 合理 |
| collect | 采集状态 | 合理 |
| lca | **状态** | 建议改为「计算状态」，与其它页「XX 状态」一致 |
| verify | 核查状态 | 合理 |

**建议**：lca 的第三块标题改为「计算状态」，便于与配置/采集/核查状态并列理解。

### 4.4 第四块：数据/统计

各页业务不同，第四块内容不同（数据项、规则数、文件数、kgCO₂e、文档数等），**可保留差异**。若需统一，可只约定「标题 + 若干 stat-item」的结构，不强制文案一致。

---

## 五、certify（报告引导页）特殊说明

- **无 task-info-dashboard**：主区为一张引导卡片，符合「仅引导去报告管理」的定位。
- **头部**：无 SLA，右侧仅头像+用户名。若希望与其它详情页「头部布局一致」，可增加 SLA 占位（如灰色「—」或「不适用」）。
- **floating-timeline**：仍存在，与主内容布局（居中卡片）并存；若此页不再强调阶段进度，可考虑收起或简化 timeline，避免与主流程四页风格差异过大。

---

## 六、其它可统一项

| 项 | 说明 |
|----|------|
| **task_detail.css 引用** | 确认 config、collect、lca、verify、certify 均引用 `../css/task_detail.css`，保证 .task-info-dashboard、.dash-block、.sla-capsule 等样式一致。 |
| **goBack 逻辑** | 各页 goBack 是否均考虑 `from=order` 时返回订单页，需核对一致。 |
| **viewOrder / viewSupplier** | 建议所有详情页都暴露 viewOrder、viewSupplier（至少与 config 一致），verify 当前缺 viewSupplier 链接。 |

---

## 七、建议的调整顺序（从配置模板开始）

1. **以 task_detail_config.html 为基准**，固定「头部结构 + 信息条四块」的 HTML 结构和用词（规格/订单/来源供应商/状态/第四块）。
2. **task_detail_collect.html**：仅做标题统一（如「目标供应商」→「来源供应商」），其余已与 config 接近。
3. **task_detail_lca.html**：补全 prod-sub（规格:、订单:、图标）；供应商标题改为「来源供应商」、副文案统一；第三块标题改为「计算状态」。
4. **task_detail_verify.html**：补全 prod-sub（规格:、订单:、图标）；供应商块改为 el-link + viewSupplier，副文案补全；确认 goBack/viewOrder 与它页一致。
5. **task_detail_certify.html**：按需补头部 SLA 占位；如需与其它页完全一致再考虑是否加简化版信息条（当前可保持仅卡片）。

按上述顺序逐页对齐后，各详情页的**头部位置布局**与**信息条结构**将一致，仅阶段名与状态/统计内容按业务区分。
