# 驳回、澄清、异议与消息功能全局规划

本文档在现有任务阶段与报告管理边界基础上，对**驳回**、**澄清**、**异议（申诉）**及**消息**进行统一规划：明确各阶段出现位置与责任方、权限与可见性规则、消息的发送/接收/消费方式，并统一报告阶段异议的处理与可见性。澄清区域 UI 与交互以 [16_澄清区域钉钉式对话设计方案](16_澄清区域钉钉式对话设计方案.md)、[17_澄清区域钉钉式对话实施方案](17_澄清区域钉钉式对话实施方案.md) 为准。

**替代关系**：本规划吸收并替代原 14_模板下发后澄清功能方案、15_消息功能规划 中与驳回、澄清、消息相关的整体设计（该两文档已删除）；实施时以本文档为准。

---

## 1. 驳回、澄清、异议：阶段、状态、责任方、处理方式

### 1.1 总览表

| 阶段 | 驳回 | 澄清 | 异议（申诉） |
| :--- | :--- | :--- | :--- |
| **采集** | 运营对供应商提交审核不通过 → 驳回；状态 `waiting_update`（整改中）；责任方：供应商整改后重交。 | 运营「提出疑点」或驳回时同步建「整改要求」类澄清；供应商在澄清中回复或按驳回原因重填。 | 不适用。 |
| **计算（LCA）** | 无独立驳回；若需退回采集则依赖任务状态回退。 | 运营/供应商就方法学、口径等澄清；运营可转派给供应商 → 子状态 `clarifying`/`waiting_supplier`，供应商待办「待澄清/待整改」。 | 不适用。 |
| **核查** | 运营对核查结果**验收驳回**（子状态 `rejected`）；责任方：核查机构重新核查或补充。 | 核查机构对**核算报告草稿**澄清/驳回 → 仅运营↔核查机构；对**供应商数据/凭证**有疑问 → 先发往运营，运营确认后决定自己回复或转发给供应商。 | 不适用。 |
| **报告** | 不适用任务内驳回；报告侧为**申诉**：供应商「我有异议」→ 运营**驳回申诉**或**接受申诉**（可回退至核查等）。 | 报告解读、确认接收前疑问可用澄清（可选）；与申诉独立。 | **异议**：供应商对已下发报告提起异议（申诉）；运营驳回或接受；接受后可回退至核查等；消息与可见性见 §3、§4。 |

### 1.2 驳回

- **采集阶段**：运营在审核中驳回 → 任务状态变为「整改中」(`waiting_update`)，可同步创建「整改要求」类澄清；供应商在待办中见「采集已驳回」，填报页可查看驳回原因并重填。
- **核查阶段**：运营对核查结果验收不通过 → 子状态 `rejected`（已驳回），责任方为核查机构。
- **报告阶段**：无「任务内驳回」；**申诉驳回**见下文「异议」。

### 1.3 澄清

- **配置（下发后）～报告**：澄清在任务已下发后可用；参与方与可见范围随阶段与类型不同（§3）。
- **核查阶段两类澄清**：  
  - **核算报告草稿**：核查机构对运营提交的草稿有疑问 → 仅运营↔核查机构，供应商不可见。  
  - **供应商数据/凭证**：核查机构发现疑问 → 先发往运营，运营确认后选择「自己直接回复」或「转发给供应商」，转发后供应商才可见并参与。

### 1.4 异议（报告阶段申诉）

- **触发**：运营将报告/证书下发至供应商后，供应商在「我的报告」中可「我有异议」提交申诉。
- **处理**：运营在报告管理档案详情中**驳回申诉**（维持原报告）或**接受申诉**（可触发回退至核查等）。
- **状态**：报告存在待处理申诉时为「申诉中」；运营处理完成后清除申诉标记，必要时回写任务状态。
- **可见性**：与澄清、驳回一致——运营端见该任务/报告下全部异议与处理结果；供应商仅见自己发起的异议及运营的驳回/接受结果；核查机构仅在业务需要时（如接受申诉后重核查）可见相关消息（§3、§4）。

## 2. 权限：谁可发起驳回 / 澄清 / 异议

| 操作 | 采集 | 计算（LCA） | 核查 | 报告 |
| :--- | :--- | :--- | :--- | :--- |
| **驳回** | 运营（驳回供应商提交） | — | 运营（验收驳回） | 运营（驳回/接受申诉） |
| **发起澄清** | 运营、供应商 | 运营、供应商 | 运营、供应商、**核查机构** | 运营、供应商（可选） |
| **回复澄清** | 运营、供应商 | 运营、供应商 | 运营、供应商（仅当参与该会话）、核查机构 | 同上 |
| **关闭澄清** | 运营 | 运营 | 运营 | 运营 |
| **提起异议（申诉）** | — | — | — | **供应商**（对已下发报告） |
| **处理异议** | — | — | — | **运营**（驳回/接受申诉） |

核查机构仅在**核查阶段**对当前任务发起澄清；对核算报告草稿的澄清仅面向运营；对供应商数据/凭证的疑问先发往运营，由运营决定是否转派给供应商。

## 3. 消息的发送、接收、消费与可见性规则（核心）

### 3.1 原则

- **运营端**：看到该任务/报告下**全部**澄清、驳回与异议相关消息（全参与方、全会话）。
- **供应商端**：仅看到**归属于供应商**的会话与消息（采集阶段运营↔供应商澄清与驳回原因；被转派的澄清；**本人发起的异议及运营处理结果**）。
- **核查机构端**：仅看到**归属于核查机构**的会话与消息（核算报告草稿仅运营↔核查机构；供应商数据类澄清在运营确认转发前仅运营↔核查机构；接受申诉后若涉及重核查，可按需展示相关消息）。

### 3.2 核查阶段两类澄清的差异化处理

1. **核算报告草稿相关**（报告内容、结论、格式等）  
   - 发起方：核查机构；接收/可见方：**仅运营**。  
   - 消息仅写入运营↔核查机构通道，不向供应商展示；运营直接回复核查机构。

2. **供应商原始数据/凭证相关**  
   - 发起方：核查机构；首接收方：**运营**。  
   - 流程：核查机构发起 → 运营端收到并**确认** → 运营选择「自己直接回复核查机构」或「转发给供应商」。  
   - 若选择转发：创建/关联一条供应商可参与的澄清会话，供应商可见并可回复；否则仅运营与核查机构可见。

### 3.3 报告阶段异议的消息与可见性

- **供应商提起异议**：生成一条「异议」类消息/记录；运营端在报告管理可见全部异议及内容；供应商端仅见自己发起的异议及状态。
- **运营驳回/接受申诉**：结果仅通知**该异议的发起方（供应商）**及运营侧；若接受并回退至核查，核查机构可见与重核查相关的任务/报告状态与消息，异议内容是否对核查机构可见由产品定（建议仅状态与结论可见，细节可仅运营可见）。
- 与澄清一致：用户级消息按「接收人」写入，仅相关方收到；协作日志在任务/报告维度记录「供应商提起异议」「运营驳回/接受申诉」，运营端可见完整流水。

### 3.4 实现要点

- 澄清会话增加**参与方/可见范围**（如 `scope` 或 `participants`：`operator_supplier` / `operator_verifier` / 待运营确认的 `operator_verifier` 后可扩展 `supplier`）。列表与详情接口按当前登录角色过滤。
- 异议（申诉）记录建议带 `reportId`/`taskId`、`initiator`（供应商）、`status`（待处理/已驳回/已接受）、处理结果与时间；消息与通知按责任方与可见性写入。
- 用户级消息（通知）按接收人写入，仅相关方收到；运营可按任务/报告维度查看全部。

## 4. 与协作日志、用户级消息的衔接

### 4.1 协作日志（任务级）

- 记录：状态变更、文件上传、澄清发起、驳回、**供应商异议（申诉）及运营处理**等。
- 条目可带 `clarificationId`、`appealId` 等，便于跳转澄清 Tab 或报告管理申诉区块。
- 运营端可见该任务下全部协作日志；供应商/核查机构仅见与己相关的摘要或入口（如「您有一条待回复澄清」），详情进入后按 §3 可见性展示。

### 4.2 用户级消息

- **类型**：任务分配、SLA 预警、澄清、驳回/整改、**报告/证书（含下发、异议、驳回申诉、接受申诉、归档）**、系统通知。
- **产生**：新建/回复/关闭澄清、驳回、供应商提起异议、运营处理异议时，向**可见范围内的相关方**写入用户级消息；跳转目标为任务详情或报告管理，澄清类可带 `?open=clarify`。
- **消费**：各端「消息中心」按当前用户过滤；运营可配置为按任务/报告查看全部或仅与自己相关的消息。

### 4.3 统一性

- 同一业务事件可既写协作日志（任务/报告维度）又写用户消息（用户维度）；消费端按角色与可见性过滤，与 §3 一致。
- 报告阶段异议与澄清、驳回采用同一套「参与方/可见范围」思路，便于扩展与运维。

## 5. 数据结构与接口建议

### 5.1 澄清会话

- **参与方/可见范围**：`scope` 或 `participants`，如 `['operator','supplier']`、`['operator','verifier']`；待运营确认的供应商数据类可为 `['operator','verifier']`，运营确认转发后扩展 `'supplier'`。
- **主题类型（可选）**：`subjectType`：`report_draft`（核算报告草稿）、`supplier_data`（供应商数据/凭证）、`general`；便于运营端展示「待确认：转派给供应商 / 自行回复」。
- **既有字段**：`taskId`、`id`、`subject`、`type`（数据完整性/凭证样式与要求/模板字段含义/方法学与计算/核查意见/整改要求/其他）、`initiator`、`initiatorName`、`status`、`stageIndex`、`messages`（含 `role`：`operator`|`supplier`|`verifier`）等，与现有 Mock 兼容。

### 5.2 异议（申诉）

- 建议字段：`appealId`、`taskId`/`reportId`、`initiator`（供应商）、`reason`、`status`（pending/rejected/accepted）、`operatorResult`、`processedAt` 等；与报告管理现有 MOCK_REPORTS 中 appeal 信息对齐。
- 列表/详情接口按当前用户过滤：供应商仅见自己发起的；运营见该报告/任务下全部。

### 5.3 接口

- **澄清**：`getClarificationsByTaskId(taskId, currentRole)`，按 `currentRole` 过滤仅返回该角色可参与的会话；运营端传 `operator` 或等价参数时返回全部。`getClarificationDetail`、`createClarification`、`replyClarification`、`closeClarification` 同前，权限与可见性由后端按任务与角色及 `scope`/`participants` 控制。
- **用户级消息**：`getMyMessages(userId, { limit, offset, unreadOnly })`；类型含澄清、驳回/整改、报告/证书（含异议）；写入时按 §3、§4 确定接收人。
- **驳回**：采集驳回原因、验收驳回、申诉驳回的存储与推送按责任方仅通知相关方（如采集驳回仅通知供应商；验收驳回仅通知核查机构；异议驳回/接受仅通知该供应商及运营）。

## 6. 各端入口（简要）

| 端 | 任务级入口 | 用户级入口 |
| :--- | :--- | :--- |
| **运营端** | 任务详情各阶段页「消息与澄清」→ Drawer（协作日志 + 澄清）；报告管理档案详情中异议处理与澄清（可选） | 消息中心（铃铛）；类型含澄清、驳回、报告/证书（含异议），跳转任务/报告 |
| **供应商端** | 待办列表、填报页、待澄清处理页的澄清入口；「我的报告」中确认接收/我有异议 | 可选消息中心；仅见归属自己的澄清、驳回、异议、报告下发等 |
| **核查端** | 核查任务详情页「澄清」入口 | 消息通知/消息中心；仅见归属核查机构的澄清与驳回 |

## 7. 实施优先级建议

| 优先级 | 内容 |
| :--- | :--- |
| **P0** | 任务级「消息与澄清」Drawer 在运营端各阶段页已实现；本规划固化阶段/状态/责任方、权限与可见性规则（含异议）。 |
| **P1** | 澄清列表/详情按角色与 `scope`/`participants` 过滤；核查两类澄清（报告草稿 vs 供应商数据）的差异化与运营确认/转派；报告阶段异议的消息与可见性按 §3、§4 实现。 |
| **P2** | 用户级消息 `getMyMessages` 及各端消息中心；新建/回复/关闭澄清、驳回、异议时写入用户消息并推送到相关方。 |
| **P3** | 协作日志与澄清/异议的联动跳转；未读/已读、红点、站内信/邮件提醒。 |
