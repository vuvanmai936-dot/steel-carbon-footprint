## 二、可信数据空间技术架构要点

### 2.1 架构组成

- **服务平台**：可信数据空间服务平台（中心化或分布式部署）。
- **接入连接器**：与服务平台对接的接入连接器，是数据流通利用的重要技术组件；需满足基本功能与安全要求，按规定的接口协议和格式适配，确保数据跨平台流通交付。

### 2.2 核心功能（技术架构中规定）

| 功能域 | 说明 |
|--------|------|
| 身份管理 | 用户/主体身份登记与认证，与「统一身份登记」对应。 |
| 连接器管理 | 接入连接器的注册、发现、状态与策略管理。 |
| 目录管理 | 数据目录、资源描述与发现，与「统一目录标识」对应。 |
| 数字合约管理 | 数字合约的创建、签署、执行与存证，支撑流程可信。 |
| 可信数据空间管理 | 空间的创建、登记、发现与逻辑空间管理。 |
| 数据使用控制 | 对数据使用行为与范围的控制与审计，支撑使用可信。 |

### 2.3 安全要求（三层）

- **数字合约安全**：合约完整性、真实性等。
- **数据产品安全**：数据分级、传输、存储与计算安全。
- **空间运行安全**：运行维护、日志存证与合规审计。

### 2.4 关键业务流程（标准中提及）

- 登记可信数据空间  
- 发现可信数据空间  
- 创建逻辑可信数据空间  
- 数据流通利用

## 三、与项目对应的技术理念（PDRD 等）

本工程（钢铁行业产业链碳足迹）中已体现或需持续对齐的要点：

- **身份可信（DID）**：用户/主体身份管理、统一身份登记，接入与认证规范。  
- **流程可信（数字合约）**：业务流程由数字合约驱动，自动执行、不可篡改，符合《可信数据空间 数字合约技术要求》。  
- **使用可信（使用控制）**：数据使用范围与方式受控、可审计，符合《可信数据空间 使用控制技术要求》（如沙箱核验等）。  
- **结果可信（VC）**：可验证凭证、存证与审计，与目录标识、运行安全相衔接。  

设计时应避免：

- 弱化或绕过「身份—连接器—目录—数字合约—使用控制」任一环节的职责边界。  
- 与「统一目录标识、统一身份登记、统一接口要求」相悖的私有化、非互操作设计。  
- 将数据使用控制仅做 UI 提示而缺少可执行、可审计的技术约束。

## 四、使用说明

1. **设计/评审时**：用本文档与 [21_与空间技术理念不一致项记录.md](21_与空间技术理念不一致项记录.md) 作为检查清单，确保功能与交互不偏离可信数据空间技术架构与建设原则。  
2. **发现不一致时**：在「不一致项记录」中新增一条，注明位置、描述、建议修正与参考标准，便于后续迭代与合规对齐。  
3. **知识库更新时**：若标准文件有更新或新增，请同步更新本文档「标准索引」与相关要点。  
4. **其他指南的对照范围**：涉及**数据资产入表与披露**（如碳足迹报告/凭证的财务确认）时，可对照《企业数据资源入表指南》；涉及**数据价值评估、定价与流通**时，可对照《数据要素价值形成规范》；涉及**数据标准分类与体系**时，可对照《国家数据标准体系建设指南》；涉及**可信数据空间标准体系与建设路径**时，可对照《可信数据空间标准化研究报告（2025 版）》。若设计与之冲突，也可在「不一致项记录」中引用上述文档。
