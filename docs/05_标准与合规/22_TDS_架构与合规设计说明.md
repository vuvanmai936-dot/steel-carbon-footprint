# TDS 架构与合规设计说明

本文档集中说明本系统在可信数据空间（TDS）/ 国家数据基础设施（NDI）中的**架构定位、身份登记与接入、目录管理与统一目录标识**，与 [20_可信数据空间标准索引与理念摘要.md](20_可信数据空间标准索引与理念摘要.md) 及 [21_与空间技术理念不一致项记录.md](21_与空间技术理念不一致项记录.md) 配套使用，供设计评审与合规对齐。

---

## 1. 架构定位与边界

### 1.1 本系统在 TDS 中的角色

本系统为**钢铁行业产业链可信数据空间**下的**业务应用层**，其后台或对接的「行业功能节点」承担**可信数据空间服务平台**能力（身份、连接器管理、目录、数字合约、使用控制等）；供应商、核查机构等 L4 角色通过**接入连接器**与服务平台对接，本应用与行业功能节点/服务平台协同，实现碳足迹核算与认证业务。

即：

- **服务平台**由行业功能节点（可信基座）提供；
- **接入连接器**由各参与方（供应商、认证机构等）或运营侧统一部署；
- 本系统主要体现为**业务节点**上的应用，与 pcf 中「行业功能节点（可信基座）与业务节点（生态市场）」的表述一致。

### 1.2 与 NDI 的层级关系

- 行业功能节点符合《数据基础设施 区域_行业功能节点技术要求》；
- 接入连接器符合《数据基础设施 接入连接器技术要求》；
- 本应用与节点/连接器的接口与数据格式需满足「统一接口要求」与互联互通基本要求。正式实施时接口与互操作以 **02 功能与对接 → [11_接口与互操作设计（实施期）.md](../02_功能与对接/11_接口与互操作设计（实施期）.md)** 及 NDI 接入连接器技术要求为准。

### 1.3 连接器—服务平台边界

- **连接器**：负责接入、策略执行、数据与日志上报；
- **服务平台**：负责身份、目录、数字合约与使用控制策略的集中管理与下发；
- **本应用**：通过行业功能节点或直接与连接器交互，完成业务操作与审计上报。

## 2. 身份登记与接入

### 2.1 职责归属

身份登记与认证在 TDS 架构中由**可信数据空间服务平台**（本场景下为行业功能节点）**统一提供或对接国家/行业身份体系**；本应用不重复建设身份源，而是**依赖空间侧统一身份登记结果**（用户/主体标识、DID、证照等），与 NDI《用户身份管理和接入规范》对齐。

### 2.2 与本应用的关系

- 运营端、供应商端、认证机构端的**登录与鉴权**在正式环境中由**空间侧或连接器**完成（如统一登录、DID 校验、U-Key 等）；
- 本应用仅消费「当前用户标识、角色、权限」等结果，用于菜单、数据过滤与审计；
- 企业信息与证照的「提交至空间」已在 [06_供应商工作台功能清单与信息结构.md](../02_功能与对接/06_供应商工作台功能清单与信息结构.md) 的 2.4 中体现，属于身份/主体数据的写入与同步。

### 2.3 正式环境身份闭环

- 接入前需在空间侧完成**主体登记**与**接入连接器注册**；
- 登录时由身份提供方校验并返回身份信息；
- 本应用按身份与策略控制功能与数据可见性，并按要求上报审计日志。

### 2.4 原型阶段说明

当前原型为演示用途，未强制登录即可访问；各端登录页（如 certifier/login.html）与 README、09 中「正式环境需接入身份与权限」的表述，均指向上述正式闭环，设计上已预留接口与职责划分。

## 3. 目录管理与统一目录标识

### 3.1 职责界定

本系统内与目录相关的职责包括：

- **数据资源/产品的登记与描述**：如任务关联的数据集、报告/证书、模板资源；
- **在空间内的发现与检索**：如按产品/任务/供应商维度的目录发现。

上述与 NDI「数据目录描述规范」「标识管理规范」的对应关系体现为**统一目录标识**。

### 3.2 设计要点

**登记**：任务实例、报告/证书、可复用模板等在应用侧生成或落库后，需在空间侧或通过连接器完成「目录登记」（资源类型、标识、描述、责任方、生命周期状态等），便于跨节点发现。

**标识**：数据资源采用统一标识体系（如与 NDI 标识管理规范一致的编码或 DID 关联），在任务、报告、模板等实体上体现唯一标识，供目录与审计使用。

**发现**：运营端/核查端按任务、订单、供应商等维度查询数据资源时，可优先从空间目录服务或连接器目录接口获取，与「业务节点」内列表形成互补；原型阶段可仍以应用内列表为主，文档中预留「目录发现」接口或扩展点。

### 3.3 实施说明

具体字段与接口在正式实施时按 NDI 数据目录描述规范与标识管理规范细化。
