# 任务澄清功能方案

本文档描述**任务级澄清**能力：在「模板配置下发成功后」首次可用，并在该任务的**配置 → 采集 → 计算 → 核查 → 报告**全流程中持续可用；参与方随阶段可能扩展（如核查阶段加入核查机构）。涵盖触发时机、参与角色、讨论主题、界面形态与数据结构，便于评审与实现对齐。

---

## 1. 目标与场景

- **目标**：为任务关联的运营、供应商及（在核查阶段）核查机构提供统一沟通通道，就模板理解、数据要求、凭证样式、方法学与核查意见等问题进行有序澄清，减少反复下发与误解。
- **场景按阶段**：同一任务下所有阶段的澄清会话**统一在一个任务级澄清列表中**，可按阶段/主题类型筛选。

| 阶段 | 澄清出现场景 | 主要参与方 |
| :--- | :--- | :--- |
| **配置（下发后）** | 模板理解、填报口径、凭证样式与要求 | 运营、供应商 |
| **采集** | 数据完整性、凭证格式、驳回后的整改说明与追问 | 运营、供应商 |
| **计算（LCA）** | 方法学、数据口径、LCA 结果说明；运营可将澄清「转派给供应商」→ 供应商待办「待澄清/待整改」 | 运营、供应商 |
| **核查** | **核查机构（如 SGS）发起澄清请求**；运营/供应商回复；收到澄清请求时运营端需高亮并提醒（与 04 一致） | 运营、供应商、**核查机构** |
| **报告** | 报告解读、确认接收前的疑问（申诉为独立流程，不替代澄清） | 运营、供应商（可选） |

---

## 2. 触发时机与可见范围

- **首次可用**：运营在「模板配置」页点击「确认配置并下发」且下发成功后，澄清入口在**运营端与供应商端**可用。
- **持续可用**：该任务在阶段 1～4 进行过程中，三端（根据角色与阶段）均可访问同一套任务级澄清列表；**核查端**在任务进入核查阶段且分配给该机构后可见并可发起/回复。
- **可见条件**：任务状态 ≥「已配置」（模板已下发）；未下发前不展示澄清入口。

澄清**应始终存在于三端**，但各端入口可见性受「任务阶段 + 角色权限」控制（例如未到核查阶段时，核查端可不展示该任务或仅只读历史）。

| 端 | 澄清入口位置 | 能力 | 可见条件 |
| :--- | :--- | :--- | :--- |
| **运营端** | 任务详情各阶段页（配置/采集/计算/核查）及报告管理档案详情（可选） | 查看列表、发起、回复、关闭澄清；接收核查机构澄清请求并高亮/提醒 | 任务已下发（已配置）及之后 |
| **供应商端** | 任务列表（待澄清项）、填报页、独立「待澄清处理」页（如 task_clarify.html） | 查看列表、发起、回复 | 任务已下发及之后；「待澄清/待整改」时突出入口 |
| **核查端** | 核查任务详情页（如 certifier/task_detail.html） | 查看列表、**发起澄清**、回复（与运营/供应商在同一会话） | 任务处于核查阶段且已分配该核查机构 |

---

## 3. 参与角色与能力

| 角色 | 能力 |
| :--- | :--- |
| **运营人员** | 查看澄清会话列表；发起新澄清主题；在任意澄清主题下回复；关闭/归档某条澄清（可选）。 |
| **供应商** | 查看澄清会话列表；发起新澄清主题；在任意澄清主题下回复。 |
| **核查机构** | 可查看该任务下的澄清列表；可**发起新澄清主题**（即文档 04 中的「澄清请求」）；可在已有主题下回复；一般不关闭澄清（由运营关闭）。 |

同一澄清会话中，消息的 `role` 需支持 `operator | supplier | verifier`，以便界面展示「运营/供应商/核查机构」身份。

---

## 4. 讨论主题（分类建议）

澄清内容按**主题**组织，便于按「数据完整性」「凭证要求」等维度检索与跟进。建议主题分类：

| 主题类型（可配置枚举） | 说明 | 示例 |
| :--- | :--- | :--- |
| **数据完整性** | 填报范围、必填项、数据口径、单位等 | “M 配料表是否需包含外购半成品？” |
| **凭证样式与要求** | 发票、检测报告、证明文件格式与内容要求 | “能耗凭证是否必须为盖章扫描件？” |
| **模板字段含义** | 对模板中某单元格/表头含义的疑问 | “‘填报指引’中‘峰谷平分项’具体指什么？” |
| **方法学与计算** | LCA 方法、数据口径、计算范围等（多出现在计算/核查阶段） | “边界取舍是否按 ISO 14067 附录？” |
| **核查意见** | 核查机构提出的疑问或要求（多由核查机构发起） | “请补充工序 A 的能耗来源说明。” |
| **其他** | 未归类的说明与沟通 | 其他协商内容 |

前端展示时可按主题类型筛选、排序；单条澄清下为「消息列表」（时间线式回复）。前端可按阶段推荐默认主题类型，或保持统一枚举由用户选择。

---

## 5. 澄清与消息的关系

- **澄清会话**：不等于「一条消息」。澄清是**带主题的多轮对话**，内含多条消息（messages），即会话本身是消息的**容器/上下文**。
- **澄清动态**：属于「消息」的一种。「新澄清发起」「某条澄清被回复」「澄清被关闭」等**事件**，应作为**一条通知/消息**推送给相关方（运营、供应商、核查机构），与「任务分配」「SLA 预警」「系统通知」等并列。文档 04 协作日志已包含「澄清发起」；新建澄清或回复时可向对方发送站内信或邮件提醒。澄清产生的动态应纳入统一「消息」体系，用户从消息入口可进入对应任务并打开澄清会话。

---

## 6. 界面形态（方案）

### 6.1 任务详情页统一消息入口（协作日志 + 澄清）

运营端任务详情各阶段页（配置/采集/计算/核查）采用**统一「任务消息」入口**，将协作日志与澄清收口到同一块区域：

- **入口**：在顶部信息条（如「配置状态」「采集状态」等）旁提供 **「消息与澄清」** 按钮（不再单独使用「澄清」按钮）。若存在**来自核查机构**或**待回复**的澄清，在入口上显示红点/角标（`hasClarifyBadge`）。
- **Drawer**：点击后打开 **「任务消息」** 侧边抽屉（约 460px），内设两个 Tab：
  - **协作日志**：展示该任务下的流水记录（状态变更、文件上传、澄清发起、驳回等），按时间排序，格式为「时间 + 用户：内容」。
  - **澄清**：即原有澄清主界面（澄清主题列表 + 选中主题的消息区 + 新建/回复）。协作日志中与澄清相关的条目（如「澄清发起」）可带链接，点击后自动切到「澄清」Tab 并定位到对应会话（可选实现）。
- **数据**：协作日志与澄清列表可来自不同接口（如现有 events/auditLogs 与 `getClarificationsByTaskId`），仅在前端做 Tab 与跳转联动。

### 6.2 各端入口

- **运营端**：在任务详情**各阶段页**（配置/采集/计算/核查），当任务已下发（配置页为 `isLocked === true`）时，在顶部信息条显示 **「消息与澄清」** 按钮；配置页仅在下发后显示。点击后打开统一「任务消息」Drawer，默认或上次 Tab（协作日志 / 澄清）均可。
- **供应商端**：在任务列表（待澄清项）、填报页、独立「待澄清处理」页（如 task_clarify.html）的显著位置提供 **「澄清」** 入口（供应商端可仍使用单一澄清入口，或后续与消息中心联动）。
- **核查端**：在核查任务详情页（如 certifier/task_detail.html）提供 **「澄清」** 入口，支持发起澄清与回复。

### 6.3 澄清主界面（Drawer 内「澄清」Tab：侧边抽屉 + 列表 + 会话）

- **布局**：采用 **侧边抽屉（Drawer）** 或 **独立弹层（Dialog）**，不离开当前任务详情页即可沟通。
- **结构**：
  1. **会话列表（左侧或顶部 Tab）**  
     - 展示当前任务下的所有「澄清主题」列表。  
     - 每条展示：主题标题、主题类型、发起人（运营/供应商/核查机构）、最后回复时间、状态（进行中/已关闭）。  
     - 支持「新建澄清」按钮，点选某条进入该主题的对话。
  2. **消息区（右侧或下方）**  
     - 选中某条澄清主题后，展示该主题下的**消息时间线**（按时间排序）。  
     - 每条消息：发言人、角色（运营/供应商/核查机构）、时间、内容（富文本或纯文本）。  
     - 底部：输入框 + 发送按钮，用于回复当前主题。
  3. **新建澄清**  
     - 表单：主题类型（下拉）、标题（简短）、首条内容（可选）。  
     - 提交后生成一条新澄清会话并进入该会话的消息区。

### 6.4 交互要点

- 新消息到达时，可在列表项上做未读标识（可选）。
- 运营端可对某条澄清标记「已关闭」，关闭后仍可查看历史，但可配置为不允许再回复（产品可定）。
- **核查机构发起的澄清**：运营端在协作日志或澄清列表中需**高亮并在合适位置弹窗/提醒**（与 04、01 一致）；澄清会话列表可标记「来自核查机构」或「待回复」。
- 若采用 Drawer，建议宽度约 460px，支持在任务详情页右侧滑出，不遮挡主内容过多。

## 7. 数据结构建议（Mock / 后端可参考）

### 7.1 澄清会话（按任务维度）

### 7.2 接口/方法建议（原型阶段可用 Mock）

| 能力 | 说明 |
| :--- | :--- |
| 按任务拉取澄清列表 | `getClarificationsByTaskId(taskId)`，返回该任务下所有澄清会话（含每条的最后一条消息摘要）。 |
| 拉取单条澄清详情 | `getClarificationDetail(taskId, clarificationId)`，返回该条澄清的完整消息列表。 |
| 新建澄清 | `createClarification(taskId, { type, subject, firstMessage })`，返回新会话。 |
| 回复 | `replyClarification(taskId, clarificationId, { content })`，在对应会话下追加一条消息。 |
| 关闭澄清（可选） | `closeClarification(taskId, clarificationId)`，将状态置为已关闭。 |

权限由后端按任务与角色过滤；消息 `role` 含 `verifier` 时，前端可展示「核查机构」身份。

## 8. 与现有流程的衔接

- **状态机**：澄清为**附加能力**，不改变任务阶段与子状态（如不单独新增「澄清中」阶段）。若产品希望在某些阶段在列表/详情中展示「存在未关闭澄清」提示，可在任务对象上增加 `hasOpenClarification: true` 等标识，由前端在配置/采集页展示角标或提示。
- **核查阶段**：若收到**核查机构发起的澄清**，运营端协作日志需**高亮并在合适位置弹窗/提醒**（与 04、01 一致）；澄清会话列表可标记「来自核查机构」或「待回复」。
- **供应商「待澄清/待整改」**：当运营或核查机构在 LCA/核查阶段将某澄清「转派给供应商」时，供应商待办中出现该任务，进入澄清页可回复并提交，与 02、06 一致。
- **权限**：仅当前任务关联的运营、供应商，以及（在核查阶段）该任务对应的核查机构可看到该任务的澄清；列表与详情按既有权限控制。
- **通知（可选）**：若后续做消息推送，可在「新建澄清」或「回复」时向对方发送站内信或邮件提醒。

## 9. 实现步骤建议（原型）

1. **数据**：在 `js/` 下新增 `mockClarifications.js`（或合并进现有 mock），实现 `getClarificationsByTaskId`、`getClarificationDetail`、`createClarification`、`replyClarification`、`closeClarification`，数据结构符合 6.1；支持 `verifier` 角色及可选 `stageIndex`/`phase`。
2. **运营端**：在任务详情**配置/采集/计算/核查**各阶段页均提供**「消息与澄清」**统一入口（配置页仅在下发后显示），打开「任务消息」Drawer，内设**协作日志**与**澄清**两个 Tab；澄清 Tab 内为「澄清列表 + 选中会话的消息区 + 新建/回复」。若有来自核查机构或待回复的澄清，入口显示红点/角标；在澄清列表中对「核查机构发起」的澄清做高亮/提醒。
3. **供应商端**：在任务列表、填报页、`task_clarify.html`（待澄清处理页）提供澄清入口，复用同一套数据结构与 Drawer/列表/消息区 UI 逻辑（可抽成共用组件或独立页面）。
4. **核查端**：在 `certifier/task_detail.html` 增加「澄清」入口，支持发起澄清与回复，消息 `role` 含 `verifier`。
5. **主题类型**：前端固定枚举「数据完整性 / 凭证样式与要求 / 模板字段含义 / 方法学与计算 / 核查意见 / 其他」，与 §4 一致；新建澄清时选择类型与标题即可。

以上为**任务澄清功能**的完整方案，可直接用于评审与分步实现。
