# 消息功能规划

本文档在现有「任务消息」（协作日志 + 澄清）基础上，明确系统中「消息」的定位、类型、任务级与用户级分层、各端入口及数据接口，形成统一的消息功能规划，并与澄清、协作日志衔接。与 [04_任务调度与状态机](04_任务调度与状态机.md)、[14_模板下发后澄清功能方案](14_模板下发后澄清功能方案.md) 保持一致。

---

## 1. 消息的定位与范围

| 层次 | 现有实现/约定 | 说明 |
| :--- | :--- | :--- |
| **任务级** | 运营端任务详情页「消息与澄清」Drawer（协作日志 Tab + 澄清 Tab） | 已实现；仅在该任务上下文内查看流水与澄清会话。 |
| **用户级** | 核查端 dashboard 的「消息通知」列表 | 含「回复了澄清」「新任务分配」「SLA 预警」等，点击跳转任务详情；运营端、供应商端无对应全局入口。 |
| **文档** | [14_模板下发后澄清功能方案](14_模板下发后澄清功能方案.md) §5 | 澄清**动态**属于消息的一种，与「任务分配」「SLA 预警」「系统通知」等并列；新建/回复澄清可发站内信或邮件。 |
| **文档** | [04_任务调度与状态机](04_任务调度与状态机.md) §3 | 协作日志记录：状态变更、文件上传、澄清发起、供应商申诉；需轮询或 WebSocket。 |

**结论**：

- **消息**在系统中有两层含义：
  1. **任务级消息**：某任务下的「协作日志」流水 + 该任务下的「澄清」会话内容（已收口到任务详情「消息与澄清」Drawer）。
  2. **用户级消息**：推送给当前登录用户的通知列表，用于「有件事需要你处理/知晓」，点击可跳转到具体业务页（任务详情、报告管理、澄清 Tab 等）。
- 规划需同时覆盖这两层，并明确类型、入口、数据与实施优先级。

## 2. 消息类型枚举（建议）

用户级消息建议按「类型」区分展示与跳转逻辑，与文档 14 和 04 对齐：

| 类型（type） | 说明 | 典型触发场景 | 跳转目标建议 |
| :--- | :--- | :--- | :--- |
| **任务分配** | 新任务被分配给当前用户/机构 | 运营创建任务、订单确认生成任务、核查任务分配 | 任务列表或对应阶段任务详情 |
| **SLA 预警** | 任务/节点即将超时 | 定时检测剩余时间 &lt; 阈值 | 对应任务详情 |
| **澄清** | 与澄清相关的事件 | 新澄清发起、某条澄清被回复、澄清被关闭 | 任务详情并可选打开「澄清」Tab（如 `?open=clarify`） |
| **驳回/整改** | 需供应商或运营跟进 | 采集驳回、LCA/核查转派供应商 | 任务详情（填报页或澄清页） |
| **报告/证书** | 报告下发、申诉、归档等 | 报告下发至供应商、申诉提交/处理、归档完成 | 报告管理档案详情或供应商「我的报告」 |
| **系统通知** | 订单确认、绑定成功、系统公告等 | 订单确认生成子任务、供应商绑定成功 | 列表或详情视业务定 |

说明：

- 澄清类消息与「任务消息」Drawer 内「澄清」Tab 对应；点击用户级「澄清」消息应进入该任务详情，并可带参数自动打开 Drawer 的澄清 Tab。
- 协作日志中的「澄清发起」等条目可与澄清会话关联，便于从日志跳转到澄清 Tab（方案 A 已预留）。

---

## 3. 任务级 vs 用户级

- **任务级**：仅当用户已进入「某任务详情」时可见；数据来源为当前任务的协作日志接口 + `getClarificationsByTaskId`；入口为任务详情页「消息与澄清」按钮 + Drawer 内协作日志/澄清 Tab。
- **用户级**：与具体任务/报告等关联，但以「当前用户待办/待阅」列表形态呈现；入口为各端「消息中心」（见下）；数据来源需规划用户维度消息接口（或从各业务事件写入消息表再按用户拉取）。
- 同一业务事件可既写入协作日志（任务维度）又写入用户消息（用户维度）。

## 4. 各端入口规划

| 端 | 任务级入口 | 用户级入口（建议） |
| :--- | :--- | :--- |
| **运营端** | 任务详情各阶段页「消息与澄清」→ Drawer（协作日志 + 澄清） | **已实现**：消息中心已覆盖所有使用标准顶栏的运营端页面，入口为顶栏铃铛图标；点击打开 Drawer 展示任务分配、SLA、澄清、驳回、报告等类型，点击列表项跳转对应任务/报告，澄清类可带 `?open=clarify` |
| **供应商端** | 待办列表「最新通知」、填报页/澄清页进入任务上下文后，可增加「消息与澄清」Drawer（与运营端一致） | 可选：顶部「消息中心」，列表含任务分配、驳回、澄清、报告下发等；跳转至待办/填报/澄清/我的报告 |
| **核查端** | 任务详情页「澄清」入口（当前已有） | 已有 dashboard「消息通知」；可统一为顶部「消息中心」入口，与运营/供应商概念一致；点击跳转 task_detail，澄清类带 `?open=clarify` |

说明：

- 任务级以现有「消息与澄清」Drawer 为主，不再新增任务内入口。
- 用户级「消息中心」为建议能力，可与现有核查端「消息通知」合并规划，实现优先级可放在任务级稳定之后。

## 5. 数据与接口建议

### 5.1 任务级（已有或已有雏形）

- **协作日志**：各详情页本地 `events` / `auditLogs`；正式实现按 04 需轮询或 WebSocket。
- **澄清**：`getClarificationsByTaskId`、`getClarificationDetail`、`createClarification`、`replyClarification`、`closeClarification`（见 14 与 js/mockClarifications.js）。

### 5.2 用户级（待规划）

- **数据结构建议**：「用户消息」表或流字段：`messageId`、`userId`（或 role+org）、`type`（任务分配/SLA/澄清/驳回/报告/系统）、`title`、`body` 或 `desc`、`relatedType`（task/report/order）、`relatedId`、`jumpUrl`（或由前端按 type+relatedId 生成）、`read`、`createdAt`。
- **接口建议**：`getMyMessages(userId, { limit, offset, unreadOnly })`；可选 `markRead(messageId)` / `markAllRead()`。
- **澄清产生用户级消息**：新建/回复/关闭澄清时，除更新任务级澄清数据外，向相关方写入一条 type=澄清 的用户消息，`relatedId=taskId`，`jumpUrl` 含 `open=clarify` 以便自动打开澄清 Tab。

## 6. 与澄清、协作日志的衔接

- **澄清**
  - 澄清会话本身 = 任务级「消息与澄清」Drawer 中的澄清 Tab 内容。
  - 澄清**动态**（新发起/回复/关闭）= 用户级消息的一种类型；若实现用户级消息中心，应在此写入并支持跳转至任务详情 + 打开澄清 Tab。
- **协作日志**
  - 保留为任务级 Drawer 的「协作日志」Tab；条目可与澄清关联（如「澄清发起」带 clarificationId），点击切到澄清 Tab 并定位会话（14 与方案 A 已描述）。
  - 协作日志与用户级消息独立：前者按任务维度、后者按用户维度；同一事件可既写协作日志又写用户消息。

## 7. 实施优先级建议

| 优先级 | 内容 |
| :--- | :--- |
| **P0（已做或已规划）** | 任务级「消息与澄清」Drawer（协作日志 + 澄清）在运营端四类任务详情页已实现；文档 14 已补充澄清与消息的关系、任务详情统一消息入口说明。 |
| **P1（建议优先）** | 运营端、供应商端任务详情（或任务上下文页）统一使用「消息与澄清」Drawer，保证三端任务级体验一致；用户级消息：定义 type 枚举与跳转规则，实现 `getMyMessages`（Mock 即可）及运营端/核查端「消息中心」入口（列表 + 点击跳转，澄清类带 `?open=clarify`）。 |
| **P2（后续迭代）** | 协作日志与澄清的联动：协作日志中「澄清发起」等可点击切到澄清 Tab 并定位；新建/回复/关闭澄清时写入用户级消息并推送到相关方；未读/已读、红点、站内信/邮件提醒（14 已约定可选）。 |
| **P3（可选）** | 供应商端顶部「消息中心」与运营/核查端统一；协作日志实时轮询或 WebSocket（04 已约定）。 |

---

## 8. 建议产出物汇总

- **产品/设计**：本文档（docs/15_消息功能规划.md）将上述类型、分层、各端入口、与澄清/协作日志的衔接、数据与接口建议、实施优先级固化，便于与 04、14 一致并指导开发。
- **前端**：在保持不修改方案 A 已实现的前提下，按优先级实现用户级消息中心入口与 Mock 接口、澄清类消息跳转带 `?open=clarify`。
- **后端/接口**：正式实现时按本规划实现用户消息表与 `getMyMessages`、业务事件写入用户消息及已读状态。

本规划不修改已确定的「任务消息」Drawer 与澄清关系，仅在此基础上补全「消息」整体范围与用户级能力，便于系统统一扩展。
