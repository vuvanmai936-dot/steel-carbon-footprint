## 二、问题清单与修改建议

### 问题 1：Sticky Footer 在带侧栏布局下占满整屏宽度

**现象**：`.sticky-footer-bar` 使用 `position: fixed; left: 0; right: 0; width: 100%`，固定相对于视口。左侧有侧栏（220px / 64px）时，底部栏会铺满整个视口，主内容区右侧的按钮与 TDS 文字会与侧栏在视觉上重叠，或显得底部栏“侵入”侧栏区域。

**建议**：
- 方案 A：让固定底部栏仅覆盖主内容区。在主内容容器（如 `el-main`）上使用 `position: relative`，将 `.sticky-footer-bar` 改为放在 `el-main` 内，并改为 `position: absolute; bottom: 0; left: 0; right: 0;`（相对 el-main 定位），这样底部栏宽度自然等于主内容区，不会盖住侧栏。
- 方案 B：若必须保持 `position: fixed`，则用 Vue 动态样式：根据 `sidebarCollapsed` 设置 `left: '220px'` 或 `left: '64px'`，并设置 `width: 'calc(100% - 220px)'` 等，使底部栏仅占主内容区域。

**推荐**：方案 A，结构更简单，无需依赖侧栏状态。

---

### 问题 2：底部操作按钮与需求「提交」不一致

**现象**：需求要求底部右侧为「通过」「驳回」「提交」三按钮，当前实现为「通过」「驳回」「提出疑点」；且审核中状态没有「提交」按钮。

**已确认**：运营端底部仅保留「通过」「驳回」「提出疑点」；提交为供应商行为，运营端不增加「提交」按钮，保持现状。

**建议**：
- 若产品确认运营端不需要「提交」按钮（提交为供应商行为），可在文档中注明“运营端底部仅保留「通过」「驳回」「提出疑点」”，并保持现状。
- 若需严格按需求增加「提交」：
  - 定义「提交」含义（例如：提交审核意见、确认提交等），在对应 `collectionState` 下在 `.sticky-footer-actions` 中增加「提交」按钮，并绑定相应方法（如 `submitAudit` 或占位方法）。

---

### 问题 3：el-page-header 的 content 插槽用法与官方结构

**现象**：当前把面包屑和产品名+状态都放在 `#content` 里。Element Plus 的 `el-page-header` 的 content 插槽通常用于替换默认的“标题”区域，部分版本下可能影响返回箭头与标题的排版（例如返回箭头与内容垂直对齐）。

**建议**：
- 查阅当前使用的 Element Plus 版本中 `el-page-header` 的 API：若支持 `#content` 仅放标题、另有插槽放面包屑，建议改为「返回 + 面包屑」一组、「产品名 + 状态 Tag」作为 content。
- 若现有排版在 Chrome/Firefox 下均正常，可仅做小调整：保证面包屑与产品名分行清晰（例如产品名加粗、与面包屑有 margin-top），避免视觉上挤在一起。

---

### 问题 4：内容区为 Sticky Footer 留白不足

**现象**：`.task-detail-scroll-body` 仅设置 `padding-bottom: 60px`。若 Sticky Footer 高度因文案换行或按钮增多而变高（例如小屏下 TDS 文案两行），60px 可能不足，最后一屏内容会被遮挡。

**建议**：
- 将 `.task-detail-scroll-body` 的 `padding-bottom` 调整为至少 `72px` 或 `80px`，或使用 `min-height` 保证底部留白足够。
- 或为 `.sticky-footer-bar` 设固定 `min-height`（如 56px），并保证 `padding-bottom` 不小于该高度 + 8px。

---

### 问题 5：澄清与日志 Tab 的 Badge 展示形式

**现象**：需求描述为「澄清与日志 (1)」，即 Tab 标签上带未读数字。当前用 `el-badge` 包住「澄清与日志」文字，未读数为 0 时通过 `:hidden="clarifyUnreadCount === 0"` 隐藏 Badge。

**建议**：
- 若产品希望与需求完全一致，可改为：Tab 的 label 显示为「澄清与日志 (n)」，n 为 `clarifyUnreadCount`，无未读时显示「澄清与日志 (0)」或仅「澄清与日志」。
- 若保持 0 不显示 Badge 的交互，可在文档中说明“0 未读时不显示数字”，视为对需求的合理优化。

---

### 问题 6：CSS 中未废弃 .task-info-dashboard 样式

**现象**：重构方案要求“废弃旧的 .task-info-dashboard 相关的 flex 样式”。当前 `task_detail.css` 中仍保留完整的 `.task-info-dashboard` 及子类样式（因其他页面如 config、lca、verify、certify、supplier 仍在使用）。

**建议**：
- 不在共享的 `task_detail.css` 中删除这些样式，否则会破坏其他详情页。
- 若仅本页（collect）不再使用，可保持现状；在文档中注明“仅 collect 页移除对 task-info-dashboard 的 HTML 使用，CSS 保留供其他页使用”。
- 若希望“废弃”仅指本页不再引用该类名，则当前已满足，无需改 CSS。

---

### 问题 7：数据采集表单 Tab 内空状态按钮文案

**现象**：空状态按钮文案为「模拟：供应商首次提交」，与 header 的「模拟提交」略有重复，且一处用冒号一处不用。

**建议**：统一文案，例如都改为「模拟：供应商首次提交」或都改为「模拟提交」，避免同义不同字。

---

### 问题 8：Sticky Footer 的 TDS 文案过长时折行

**现象**：TDS 声明较长，小屏或窄容器下会折行，可能导致底部栏变高、与「左侧 TDS、右侧按钮」同一行的设计错位。

**建议**：
- 为 `.sticky-footer-tds` 设置 `flex: 1; min-width: 0;`（已有），并加 `line-height: 1.5`，必要时 `max-height` + `overflow-y: auto` 或限制为最多两行并 `line-clamp`。
- 或缩短文案为：“可信数据空间：本任务数据由数字合约保护，全程哈希存证。”

## 四、可直接交给 Gemini 的执行摘要

请按以下顺序修改 `operator/task_detail_collect.html` 与 `css/task_detail.css`：

1. **Sticky Footer 仅覆盖主内容区**  
   - 在 `el-main` 上增加 `position: relative`。  
   - 将 `.sticky-footer-bar` 的定位改为 `position: absolute; bottom: 0; left: 0; right: 0;`，并确保该 div 仍在 `el-main` 内、与 `task-detail-scroll-body` 平级。  
   - 这样底部栏宽度等于主内容区，不再盖住左侧侧栏。

2. **增大内容区底部留白**  
   - 在 `task_detail.css` 中把 `.task-detail-scroll-body` 的 `padding-bottom` 从 `60px` 改为 `72px` 或 `80px`。

3. **底部按钮与需求对齐**  
   - 若需求明确要求「提交」：在 `collectionState === 'auditing'` 等合适状态下，在 `.sticky-footer-actions` 中增加「提交」按钮并绑定方法。  
   - 若需求为「通过 / 驳回 / 提出疑点」，保持现状并在文档中说明。

4. **澄清与日志 Tab**  
   - 若需 Tab 标签显示为「澄清与日志 (n)」：用自定义 label 插槽，内容为 `澄清与日志 ({{ clarifyUnreadCount }})`，或继续用 el-badge 并确认 0 时是否显示「(0)」。

5. **文案统一**  
   - 将 header 的「模拟提交」与数据采集表单 Tab 内的「模拟：供应商首次提交」统一为同一套文案（二选一或统一为一种说法）。

6. **CSS 与其它页**  
   - 不删除 `task_detail.css` 中 `.task-info-dashboard` 等样式，供其他详情页使用；仅在本页不引用对应类名即可。

完成以上修改后，在多种分辨率下检查：中间内容区可滚动、底部栏固定在主内容区底部不遮挡侧栏、最后一行内容不被遮挡。
