# ADR-0001: 设计 SoT × 可演示原型 × 开发实现 三层分离

- **状态**：Accepted
- **日期**：2026-05-09
- **作者**：产品经理（孙抗峰）
- **影响仓**：原型仓 0209、`pcf-center-fe / back`、`pcf-supplier-fe / back`、`pcf-audit-fe / back`

## 1. 背景

0209 仓库自起到现在累积了：① 60+ 多页静态页面（三端工作台）、② `docs/` 五个子目录的设计文档、③ `客户商务/` 的报价/方案材料、④ 与内网 GitLab 组 [`blockchain-future`](http://192.168.11.226/blockchain-future) 下六个开发仓的非正式对应关系。问题是"三种东西混在同一仓库 + 同一心智模型里"：

- 文档反复出现"现行 vs 历史"模糊（如 `08_PR_…` vs `04 状态机`、`16/17` vs `18/19`）。
- 原型与开发实现没有契约管道，只有"截图 + 微信对齐"。
- 评审/演示要在 60+ 页里挑路径，非常慢且容易踩到分支页。
- 任何一处改动都要联动多 HTML + Mock + docs，**返工成本高**，是产品经理"不满意"的最直接来源。

之前的方案（PR 长文 + 文档堆积）已被证明无法收敛。

## 2. 决策

把仓库与组织模式按以下三层分离，并以 `docs/SoT_MAP.md` 为单一权威映射：

```
设计 SoT（业务规则 / 状态机 / 字段词典 / OpenAPI 契约）
   └─→ 可演示原型（5 分钟黄金动线 + 关键页 + SpreadJS 演示）
   └─→ 开发实现（六仓 fe/back，自治；从 SoT 消费契约，PR 反馈差异）
```

**强约束**：

- 任何业务/字段/状态变更，**先改 SoT 文档** → **再在 [`docs/00_迭代计划/设计变更记录.md`](../00_迭代计划/设计变更记录.md) 登记一行** → **再通知开发仓**。
- 开发仓发现 SoT 错误，开 ADR 或在差异清单里记录，回写 SoT；**不在六仓里"另起炉灶"**。
- 历史方案归档至 [`docs/_archive/`](../_archive/)，不再作为现行依据。

## 3. 备选方案

| 方案 | 优点 | 缺点 | 不选原因 |
|------|------|------|----------|
| A. 维持现状（多页全功能复刻 + 设计文档堆叠） | 短期改动 0；任何想得到的页面都"有" | 心智成本越来越高，返工成本指数上升 | 已成为不满意的根因 |
| B. 把原型直接迁入六仓的 fe，删掉 0209 仓 | 一步到位，与开发同源 | 损失原型作为"评审/对外演示"的灵活性；PM 失去"试笔本"；六仓 fe 节奏与设计节奏不一致 | 损失太大；六仓 fe 不应承担方案探索 |
| C. 三层分离 + 单一 SoT 映射（本决策） | 各层职责清晰，演进可独立；PM 仅需维护设计 SoT 与一条黄金动线 | 需要一次性建立 `_archive/`、`SoT_MAP.md`、OpenAPI、ADR、差异清单等基础设施 | **采纳** |
| D. 把 `docs/` 整个迁到 Wiki/Notion | 编辑体验好 | 失去 git history 与 PR review；与原型/开发仓割裂 | 不选，仓内归档反而更可控 |

- **原型仓变更（已落地）**：物理删除 `operator/task_detail.html`；归档 7 份历史文档与 1 份 refactor 示例至 [`_archive/`](../_archive/)；新增 [`prototype/flows/golden_5min.html`](../../prototype/flows/golden_5min.html) 作为 5 分钟黄金动线；layout 双份维护改为构建期单向同步。
- **代码/工程影响（待开发对齐）**：六仓 fe/back 在第一次对齐时按 [`六仓实现差异清单_模板`](../00_迭代计划/六仓实现差异清单_模板.md) 走查；OpenAPI §7 字段以 SoT 为准。
- **不在范围**：本 ADR 不决定六仓的内部技术栈（前端框架/后端语言）、不决定 SpreadJS 商业授权（见 [ADR-0002](ADR-0002-spreadjs-license-and-sfc-migration.md)）。
