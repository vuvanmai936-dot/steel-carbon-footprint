# Diátaxis 文档分类约定

> **目的**：让 `docs/` 下的每篇文档归属唯一一个 Diátaxis 类别，避免再出现"既像方案又像审查又像 SOP"的混合长文。新写文档前先选类别，再动笔。

## 四个类别（Diátaxis）

| 类别 | 一句话 | 给谁看 | 现仓示例 |
|------|--------|--------|----------|
| **Tutorial（教程）** | 跟着我做一遍，你会得到一个能跑起来的东西 | 新人 / 第一次接触系统的人 | [`prototype/flows/README.md`](../prototype/flows/README.md) 5 分钟动线脚本 |
| **How-to（操作指南）** | 已经知道目标，告诉我步骤 | 在岗成员 / 维护者 | [`docs/00_迭代计划/六仓实现差异清单_模板.md`](00_迭代计划/六仓实现差异清单_模板.md)、[`docs/_archive/README.md`](_archive/README.md)（如何归档） |
| **Reference（参考）** | 我要查具体字段/枚举/接口 | 写代码/查规则的人 | [`docs/01_业务与流程/02_全局数据字典与枚举.md`](01_业务与流程/02_全局数据字典与枚举.md)、[`docs/02_功能与对接/11_接口与互操作设计（实施期）.md` §7](02_功能与对接/11_接口与互操作设计（实施期）.md)、[`docs/SoT_MAP.md`](SoT_MAP.md) |
| **Explanation（解释）** | 为什么是这样？背景与权衡 | 评审/新加入的产品/技术 | [`docs/01_业务与流程/01_整体业务与产品设计审查.md`](01_业务与流程/01_整体业务与产品设计审查.md)、[`docs/05_标准与合规/22_TDS_架构与合规设计说明.md`](05_标准与合规/22_TDS_架构与合规设计说明.md)、ADR |

## 写之前问自己 3 个问题

1. **谁看？**——新人 / 在岗 / 写代码 / 评审。  
2. **看完拿走什么？**——能跑通 / 能查到 / 能讨论决策 / 能照着做事。  
3. **此刻文档库里已有同类的吗？**——若有，先尝试在已有文档里追加章节，而非新开文档。

> 如果两个类别都像，则**拆成两份**：操作步骤进 How-to，背后原理进 Explanation；不要写"既是方案又是 SOP"的混合体（这是历史归档目录里大多数文档的死因）。

## 与 ADR / 设计变更记录 的关系

- **ADR**：解释架构决策（Explanation），不写步骤、也不写字段词典。  
- **设计变更记录**：纯 Reference（一行一变更）。  
- **走查清单 / 差异清单**：纯 How-to（按行操作）。
